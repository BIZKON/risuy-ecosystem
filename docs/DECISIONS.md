# DECISIONS — журнал необратимых/архитектурных решений reseller-платформы

Формат: дата · решение · почему · что бы его отменило.
Сюда пишутся решения, на которые ссылается `docs/reseller-platform-tz.md` (§2, §9).

---

## 2026-06-12 — Базовые решения адаптации ТЗ (v1.0-risuy)

1. **Деньги в новых таблицах — только целые µRUB (`bigint`, 1 RUB = 1 000 000 µRUB).**
   Почему: float-дрейф на деньгах недопустим; целочисленная арифметика ассертится
   в приёмке. Существующие `numeric(12,2)` (orders, service_invoices) — legacy,
   конверсия на границе. Отменило бы: ничего (необратимое по определению).

2. **Множитель наценки и цена сообщения — только сервер (`plans`), снимок пишется
   в каждую строку леджера.** Почему: клиентский запрос не должен уметь влиять
   на маржу. Отменило бы: ничего.

3. **Округление денег — всегда вверх (`ceil_mul`), одна функция `shared/money.py`.**
   Почему: округление в нашу пользу, единая точка против дублей-расхождений.

4. **Метеринг v1 — свой append-only леджер в Postgres, без Lago/OpenMeter.**
   Почему: отдельный метеринг-сервис — лишняя операционная нагрузка при текущем
   масштабе (1 тенант, цель 3–5). Порог пересмотра: >50 тенантов ИЛИ >1М строк
   леджера/мес ИЛИ потребность в realtime-агрегатах, которые Postgres не тянет.

5. **Метеринг cloud-ai — по дельте `used_tokens` агента (снапшоты), НЕ оценкой
   по длине текста.** Почему: `/call` не возвращает usage (проверено в проде
   2026-06-12); агент принадлежит ровно одному тенанту → дельта точна.
   Разбивка вход/выход недоступна → смешанная цена (доля выхода — конфиг, 0.5
   для thinking). Отменило бы: появление usage в ответе `/call` (тогда —
   per-call метеринг, снапшоты остаются сверкой).

6. **Мультиплекс ботов v1 — polling-таски в одном процессе (app 201859).**
   Почему: нулевая инфра-перестройка, hot-reload по реестру тенантов.
   Порог пересмотра (переход на вебхуки): >20–30 активных тенантов, либо
   деградация латентности ответа, либо лимиты Telegram на polling.

7. **Tenant-изоляция — общая БД + `tenant_id` + Postgres RLS
   (`current_setting('app.tenant_id')`), роль panel_rw без bypassrls.**
   Почему: «инстанс на клиента» (клон приложений Timeweb) дороже (~510 ₽/мес × N)
   и не даёт платформенной сводки; RLS — стандартный механизм shared-schema.
   Отменило бы: требование клиента на физическую изоляцию (тогда — выделенный
   инстанс как премиум-опция поверх той же кодовой базы).

8. **Первый тенант = «Школа Лесова», создаётся backfill-миграцией текущих данных.**
   Почему: прод не останавливается; критерий «Школа не сломана» (§8.7) — в каждой
   волне. Отменило бы: ничего.

9. **Better Auth / TS-SDK ЮKassa из оригинального ТЗ НЕ используются.**
   Почему: проект на Python; своя auth (argon2, lockout-гарантии) и свой
   `yookassa.py` (чеки 54-ФЗ, два магазина, вебхук с re-fetch) уже в проде.

10. **KikuAI Masker не обязателен для v1.** Почему: оба LLM-бэкенда — РФ-инфра
    Timeweb, трансграничной передачи ПДн нет. Становится обязательным звеном
    ДО вызова при подключении любого зарубежного провайдера (отдельная волна).

11. **Dev-БД = `risuy_dev` на том же кластере 4171827.** Почему: у Timeweb DBaaS
    нет branching; отдельная база — 0 ₽ и достаточно для прогона миграций до прода.

## 2026-06-12 — Wave 0 выполнена (DDL в проде)

12. **`tenant_id DEFAULT <uuid Школы>` на всех 12 существующих таблицах —
    переходник до Wave 3.** Почему: живой код (бот/панель) вставляет без
    tenant_id; NOT NULL без default ломал бы вставки прямо в проде (поймано при
    выкатке, окно ~3 мин, пострадавших вставок не было — проверено по messages).
    Снимается миграцией Wave 3, когда мультиплекс пишет tenant_id явно.

13. **RLS на СУЩЕСТВУЮЩИХ таблицах в Wave 0 НЕ включён** (только на 7 новых
    tenant-scoped). Почему: текущая панель не выставляет `app.tenant_id` —
    включение спрятало бы данные Школы (слом §8.7). Включение — Wave 1, отдельной
    миграцией, после деплоя tenant-context middleware. До этого изоляция
    существующих данных — приложением (как сейчас).

14. **Расширения и владелец dev-базы управляются через API Timeweb, UI не нужен**
    (находка Wave 0, уточняет старую заметку «только UI»):
    `PATCH /databases/{cluster}/instances/{id}` принимает
    `{"extensions":{"extension_list":[...]}}` и `{"owner_id":<admin_id>}`.
    База, созданная через API, по умолчанию принадлежит системному владельцу и
    не даёт CREATE в schema public — назначить owner_id (gen_user = 404579).

15. **Прогон миграций на dev = ВЕСЬ стек с нуля (16 файлов)**, не только новые.
    Почему: честный тест воспроизводимости схемы; поймал отсутствие `created_at`
    у link_clicks/broadcast_recipients до прода.

## 2026-06-13 — Wave 3 (metering + cost-capture, код в проде)

16. **Gateway cost-capture — AT-MOST-ONCE в v1 (без durable-outbox).** Списание за
    gateway-вызов (`api.timeweb.ai`) идёт fire-and-forget после ответа лиду;
    `idempotence_key=gw:{tenant}:{request_id}` страхует от ДУБЛЯ, но не гарантирует
    доставку — при рестарте/недоступности БД в момент ответа списание теряется
    невосстановимо (usage живёт только в памяти; снапшоты used_tokens покрывают
    cloud-ai, у gateway свой счётчик). Почему допустимо: дефолтный бэкенд —
    cloud_ai (метрируется снапшотами по дельте), gateway включается опционально.
    Порог пересмотра (durable-outbox: запись usage в таблицу в том же процессе
    ответа + сверка по ней): перевод prepaid-тенантов с реальными кошельками на
    gateway-бэкенд. Отменило бы: появление usage в ответе cloud-ai `/call`
    (тогда оба бэкенда метрируются единообразно по ответу).

17. **per_message: `cost_microrub=0` в леджере (charged = цена сообщения плана).**
    Почему: разбивки токенов на сообщение нет, а себестоимость per_message-тенанта
    контролируется отдельно — снапшотами used_tokens агента как СВЕРКОЙ (пишутся,
    но не списываются). `charged` = `per_message_microrub` плана (сервер). Первый
    скан тенанта = baseline (`metering_msg_hwm = max(id)` истории Лии без списания),
    иначе вся история затарифицировалась бы задним числом.

18. **Тенант по умолчанию (Школа) НЕ блокируется метерингом никогда, даже получив
    план.** Почему: §8.7 — бот Школы не должен замолчать ни на минуту, а до Wave 4
    (миграция Школы на subscriptions) назначение ей плана сделало бы `prepaid=True`
    и при накопленном минусе кошелька включило бы `ai_wallet_blocked` для ВСЕХ её
    лидов. Защита: `_maybe_block_wallet`/cost-capture сверяют `tenant ==
    default_tenant_id()` и вместо паузы шлют ops-алерт. Снимается, когда у Школы
    появится реальный кошелёк с топапами (Wave 4).

## 2026-06-13 — Wave 4 (связка оплаты подписки с метерингом, код в проде)

19. **Оплата тарифа связана с метерингом через НОВЫЙ `metadata.kind='platform_subscription'`,
    legacy-ветка вебхука сохранена.** Почему: боевая оплата Школы 3750₽ идёт по
    legacy-ветке (`mark_service_invoice_paid_by_payment`) — её нельзя ломать. Новая
    ветка вебхука (между topup и legacy) при `kind=platform_subscription` делает ОБА:
    legacy mark paid (UI-витрина `/subscription` из `service_invoices`) + активация
    `subscriptions` + начисление `included_credits` (источник тарифа для метеринга).
    Платежи без нового kind (старые/ручные/публичный лендинг) падают в legacy-фолбэк
    как раньше. Две системы (`service_invoices` legacy-витрина ‖ `subscriptions`
    метеринг) сосуществуют — без DDL, `plan_key`==`plans.code`.

20. **`included_credits` начисляются из ПЛАНА, идемпотентно по `yookassa_payment_id`
    (payments-журнал).** Почему: квота тарифа (econom=3750₽=500 сообщений) — фикс
    плана, НЕ из суммы платежа (overage за прошлый период в кредиты не идёт).
    Идемпотентность — `payments(type='subscription')` с `unique(yookassa_payment_id)`:
    повторный вебхук → INSERT ON CONFLICT DO NOTHING → no-op (как `mark_topup_succeeded`).
    ⚠️ E2E покупки через ЮKassa не прогнан live (магазин в env — БОЕВОЙ 1378536, не
    тестовый 1379463): связка проверена смоуком на risuy_dev (11/11) + подтвердится
    автоматически при первой реальной оплате подписки Школой (решение владельца).
    Каждая оплата = новая `subscriptions`-строка active на 30 дней; `get_tenant_plan`
    берёт latest active. Рекуррент-автосписания (Wave 2b) ждут менеджера ЮKassa.

21. **Мультиплекс v1 (Wave 3e) — known-ограничения, не блокеры.** (а) `leads.tg_user_id`
    глобально уникален → один Telegram-пользователь, написавший ботам ДВУХ тенантов,
    делит один `leads`-ряд. Не критично при текущем масштабе (1 тенант, цель 3–5);
    для воронки тенанта v2 — сделать лид уникальным по `(tenant_id, tg_user_id)`.
    (б) Тенант-бот v1 = «голая Лия» (AI + метеринг); полная воронка (гейт-канал,
    выдача гайда, продукты, рассылки) требует per-tenant config — следующая подволна.
    (в) Ревью 3e через адверсариальный workflow дважды зависло на стадии чтения
    (нестабильность среды) → проведён self-review: критичных багов нет, изоляция §8.7
    подтверждена в проде (пустой реестр = no-op, бот Школы отвечает).
