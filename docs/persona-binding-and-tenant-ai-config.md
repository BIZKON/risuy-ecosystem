# Дизайн: надёжное применение инструкций ИИ — персона Лии (Школа) + per-tenant конфиг клиентов

Контекст: инструкции, заданные в панели, не доходят до бота, потому что путь
«конфиг → промпт в вызове LLM» неполон. Документ фиксирует ДВА фикса:
A. Школа/Лия — сделать так, чтобы персона применялась ко ВСЕМ диалогам (выбор владельца).
B. Клиенты (reseller) — per-tenant раздел инструкций, которого сейчас НЕТ.

---

## A. Лия (Школа-бот): глобальная активная персона

### Как сейчас (проверено по коду)
- `handlers.on_free_text` → `lead_persona = get_lead_persona(tg_id)` (это `leads.ai_persona`,
  выбор оператора на КОНКРЕТНЫЙ диалог) → `get_ai_overrides(source, lead_persona)`.
- `get_ai_overrides` берёт промпт по приоритету: **персона диалога** (`ai_persona_prompt__<persona>`)
  → **канал** (`ai_system_prompt__<source>`) → **глобальный** (`ai_system_prompt`).
- Глобальное поле «активная персона» (`app_settings.ai_persona`, его пишет /agents) **бот
  игнорирует** — оно используется только для витрины/ярлыка в панели. → даже «выбрав персону
  Лия» в /agents, ты не влияешь на то, какой промпт уходит в вызов.
- Итог: персона `liya` применяется ТОЛЬКО если оператор вручную проставил её на диалог.
  Не проставлено → глобальный пустой → агент на заводском промпте.

### Фикс (код, маленький и точечный)
Научить бота чтить **глобальную активную персону** как фолбэк:
- `get_ai_overrides(source, persona)`: если `persona` (из диалога) пуст И нет канальной
  привязки → подставить глобальную активную персону `app_settings.ai_persona` и взять
  `ai_persona_prompt__<active>` / `ai_persona_agent__<active>`.
- Приоритет остаётся: **диалог > канал > глобальная персона > глобальный промпт**.
- В `handlers.on_free_text` (и для единообразия — multiplex) пробрасывать активную персону,
  если у лида своя не выбрана.

После фикса: достаточно в «ИИ-агенты» выбрать активную персону «Лия» (`ai_persona='liya'`) —
её промпт (`ai_persona_prompt__liya`, уже заполнен на 20k) применится ко ВСЕМ диалогам Школы.

### Проверка (канарейка — докажем доставку, не на словах)
В конец промпта Лии добавить временное правило:
«Если сообщение клиента РОВНО `пинг-лия-настройка`, ответь РОВНО `Промпт активен ✅ (SPIN)` и
больше ничего.»
Владелец пишет боту `пинг-лия-настройка` → если пришёл `Промпт активен ✅ (SPIN)` — промпт
реально доехал и применяется. Если обычный ответ — значит срабатывает фолбэк на нативный
`/call` (OpenAI-эндпоинт агента недоступен) → чиним `TIMEWEB_AI_OPENAI_BASE` / серверный
промпт агента. После проверки правило-канарейку убрать.

### Риск-заметка (механизм №2, держать в уме)
Даже при правильной персоне промпт теряется, если OpenAI-эндпоинт агента недоступен и бот
падает на нативный `/call` (он промпт не передаёт — агент отвечает серверным промптом).
Канарейка это и выявит. Запасной, самый надёжный для cloud_ai вариант — продублировать
промпт в серверный промпт агента («Базы знаний»), тогда он применяется на ОБОИХ путях.

## B. Клиенты (reseller): per-tenant раздел инструкций

### Как сейчас (проверено по коду) — пробел
- Клиентские tenant-боты (multiplex) читают промпт из `get_tenant_ai_overrides(tid)` →
  `tenant_settings.ai_system_prompt` (один промпт на тенанта, без слоёв персона/канал).
- Панельный раздел «ИИ-агенты» (`set_ai_settings`) пишет **глобальный `app_settings`** (Школы),
  НЕ `tenant_settings`. Per-tenant UI инструкций нет.
- ⇒ Клиент, отредактировав «ИИ-агенты», (а) изменил бы ГЛОБАЛЬНЫЕ настройки Школы (кросс-тенант!),
  (б) его бот всё равно читает свой `tenant_settings` → инструкция не применилась бы.
  Сейчас провижининг клиента (агент + промпт) — ручной (см. handoff).

### Дизайн per-tenant конфига
1. **Хранилище:** `tenant_settings` (уже есть, RLS tenant-scoped). Ключи — зеркало app_settings:
   `ai_enabled`, `ai_backend`, `ai_agent_id`, `ai_system_prompt`, `ai_fallback_text`
   (+ позже `ai_persona`/`ai_persona_prompt__<slug>` для персон клиента).
2. **Запись:** новый раздел панели «Мой ИИ-сотрудник» (для роли operator — СВОЙ тенант),
   `set_tenant_ai_settings(tid, ...)` пишет `tenant_settings` под `set_config('app.tenant_id')`
   (как остальные tenant-мутации). Гейт: operator пишет ТОЛЬКО свой активный тенант
   (actor==owner membership); платформенный супер может править любой через /tenants-switch.
3. **Чтение:** `get_tenant_ai_overrides` уже читает `tenant_settings.ai_system_prompt` — менять
   не нужно. Лимит промпта поднять до cloud_ai-уровня (как агент, ~20k), не 4000.
4. **Агент тенанта:** каждому платящему клиенту — свой cloud-ai агент (его access_id →
   `tenant_settings.ai_agent_id`); промпт пушится и на серверный промпт агента (надёжность
   на обоих путях). Создание агента — платная операция Timeweb → провижининг владельцем
   (как сейчас), но ПРОМПТ клиент правит сам в своём разделе.
5. **Изоляция:** `tenant_settings` под RLS; клиент видит/пишет только свой; TIMEWEB-токен —
   в env, клиенту не виден (white-label: модель не светим, бренд «ИИ-Агент Про»).

### Объём (оценка)
- Схема: ключи в `tenant_settings` (уже существует) — DDL не нужен, только новые ключи.
- Панель: раздел «Мой ИИ-сотрудник» (GET+POST) + `set_tenant_ai_settings` (tenant-scoped).
- Бот: уже читает `tenant_settings.ai_system_prompt` (multiplex) — без изменений; опц. поднять
  доставку на серверный промпт агента при провижининге.
- Смоук: 2 тенанта, каждый видит/пишет только свой промпт (RLS).
