# 🔴 ПРИОРИТЕТНЫЙ ПЛАН #2 (пересмотр) — приватный РФ-LLM-бэкенд: self-host Gemma (претендент №1)

> **Статус:** ПРИОРИТЕТ в очереди 152-ФЗ (после `#1` обезличивание LIVE `2f3110e` и продукт-фундамента
> СП-1 «ИИ-команда» LIVE `da5f54e`). Дизайн модели ВЫБРАН (Gemma #1, решение владельца 2026-06-29);
> интеграция и параметры — детализировать перед реализацией. Заменяет `2026-06-28-yandexgpt-direct-backend.md`.
> Часть 152-ФЗ: `#1 выгрузка ✅` → **`#2` РФ-резидентный бэкенд (этот док)** → `#3` «Справка о ПД».

## Зачем (две проблемы разом)
1. **152-ФЗ / трансгран.** Текущий путь Timeweb AI Gateway → DeepSeek гонит ПДн через юр.лицо Timeweb в
   Казахстане (офиц. ответ Timeweb, `docs/152fz-legal-review-checklist.md` C.1) → декларация «трансграна нет»
   ложна. Нужен РФ-резидентный инференс.
2. **🔴 КОНТЕКСТНОЕ ГОРЛЫШКО (вскрыто владельцем 2026-06-29).** Системный промпт Лии — до **20k символов**
   (`TENANT_AI_PROMPT_MAX`); YandexGPT на Timeweb даёт **32k вход / выход не указан** и слабо держит длинный
   промпт + длинные многооборотные сессии. Это убивает качество. Нужен **большой контекст + строгое следование
   длинному промпту**.

## Решение (владелец, 2026-06-29): self-host Gemma #1
- **Модель:** **Gemma 4** (релиз 02.04.2026) — **Apache 2.0**, **контекст до 256k** (12B/26B-MoE/31B; E2B/E4B —
  128k), **нативный function-calling** (спецтокены `<|tool_call>`/`<|tool_response>` + JSON-schema). Это прямо
  закрывает и горлышко контекста (256k ≫ 32k), и tool-calling, и лицензионную чистоту. Русский — по оценке
  владельца «норм» (сверить эвалом на промпте Лии до прод). Запас: Gemma 3 27B (128k, кастомная лицензия) —
  только как дешёвый порог входа.
- **Хостинг:** РФ-GPU (аренда), **vLLM** → OpenAI-совместимый `/v1/chat/completions` с tool-calling.
- **152-ФЗ:** self-host в РФ-ЦОД = инференс и данные в РФ, **нет трансграна, нет стороннего обработчика → DPA
  не нужен** (risuy сам обработчик). Самый чистый путь (особенно для multi-tenant: тенантам не нужен DPA с
  внешним API). Митигирует и оборотные штрафы за трансгран (до 18 млн ₽ повтор).

## Интеграция (МОДЕЛЬ-АГНОСТИЧНА — переиспользует существующее)
self-host Gemma через vLLM = **OpenAI-совместимый эндпоинт**, поэтому встаёт в уже существующий механизм:
- `bot-telegram/ai.py::ask_gateway` (OpenAI-совм., mask→send→unmask + usage-метеринг) — нацелить на РФ-vLLM
  через `provider=custom`-стиль: `gateway_base_url` = `https://<rf-gpu-host>/v1`, `model` = тег Gemma, ключ.
  Либо новый `backend='selfhost'` в `_ask_ai_backend` (ai.py:399) — копия gateway-ветки с другим base_url/токеном.
- **Тумблер в админке** Timeweb-gateway ↔ РФ-self-host (config-only через `tenant_settings`/`app_settings`
  `ai_backend`/`ai_gateway_base_url`). YandexGPT/GigaChat — оставить как запасной managed (демотированы).
- PII-маскировка (`shared/pii`) сохраняется (defense-in-depth).
- ⚠️ **Метеринг**: self-host — фикс-стоимость GPU/мес, не per-token; gateway-usage-учёт неприменим → метеринг
  тенантам пересмотреть (по сообщениям/флэт) для self-host-бэкенда.

## Параметры железа/контекста (из исследования)
- Gemma 4 **31B Q4 ~17.5 ГБ**, **26B-MoE Q4 ~14.4 ГБ** (активных ~4B → быстрее), 12B Q4 ~6.7 ГБ. Gemma 3 27B
  QAT int4 ~14 ГБ. → влезает на **одну 24ГБ-карту** (RTX 3090/4090/A5000).
- ⚠️ **Контекст vs VRAM**: полный 256k KV-cache не влезет на 24ГБ → практично **эффективный контекст 32–64k на
  24ГБ** (под 20k-промпт Лии ≈ ~8–10k токенов + длинная история — с запасом); полный 128–256k → карта 48–80 ГБ.
- РФ-GPU аренда (открытый прайс immers.cloud/Intelion): **RTX 3090 ~37–68 ₽/час (~27–49к ₽/мес)**, RTX 4090
  ~84 ₽/час (~60к ₽/мес); A100 80ГБ ~160–255к ₽/мес. Timeweb/Selectel/Cloud.ru — карты есть, цена за заявкой.
- ⚠️ **Дефицит GPU в РФ 2026** (санкции на A100/H100): аренда у РФ-провайдера снимает закупку, но ставки высоки
  и возможна нехватка; одна карта = SPOF (нужен резерв/managed-фолбэк).

## 🔴 Открытые вопросы (закрыть до реализации)
- **Timeweb-managed Gemma** (их предложение): какая версия (4?), размер, квантизация, **реальный контекст
  вход/выход**, **где физически GPU (РФ?)**, изоляция/нехранение. Managed-Gemma годится по 152-ФЗ только если
  инференс физически в РФ; turnkey без своего DevOps — плюс, но проверить контекст (не 32k!).
- **Self-host vs Timeweb-managed Gemma**: своя vLLM-инсталляция (полный контроль контекста/версии) vs managed
  (без DevOps, но зависимость от их конфига). Выбрать после ответа Timeweb по контексту.
- **Эвал русского + следование 20k-промпту Лии** на Gemma 4 (31B vs 26B-MoE) на длинной сессии — ДО прод.
- **vLLM tool-calling на reasoning-моделях**: Gemma 4 форсит thinking → нужно `--jinja
  --chat-template-kwargs '{"enable_thinking":false}'` + tool-call-parser; запинить версию vLLM (известные баги
  парсеров). Прогнать агентный цикл, не одиночный вызов (как с DeepSeek-flash на gateway).
- **Метеринг** для фикс-costs self-host (флэт/по сообщениям вместо per-token).
- **Бюджет/SPOF**: ~30–60к ₽/мес GPU + DevOps; при 100–500 диал/день self-host дороже managed (окупаемость
  ~2000 диал/день) → решение compliance-driven, не экономикой. Резерв/фолбэк на managed на случай падения карты.

## Архитектурный рычаг (связь с СП-2)
«Бесконечная память на RAG» (roadmap СП-2) СНИМАЕТ раздувание промпта: знания не пихаем в 20k системного
промпта, а достаём retrieval'ом → меньше промпт, дольше держится сессия. Т.е. контекстное горлышко лечится и
моделью (большой контекст Gemma), и архитектурой (RAG). Два рычага складываются.

## Запасные варианты (если Gemma не зайдёт по русскому/tool-calling на эвале)
- **Qwen3-32B** (128k, Apache 2.0, сильный tool-calling) / **T-pro 2.0** (Т-Банк, рус-дотюн Qwen3-32B, лучший
  русский среди open, 128k, Apache 2.0) — self-host, тот же vLLM-стек.
- **YandexGPT / GigaChat** (managed РФ) — быстрый старт без железа, НО: 32k на Timeweb (горлышко!), сторонний
  обработчик (нужен enterprise-DPA + согласие; стандартные оферты GigaChat запрещает ПДн / YandexGPT резервирует
  права). Только запасной.
