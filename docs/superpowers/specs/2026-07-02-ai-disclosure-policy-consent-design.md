## 1. Контекст и проблема

`shared/leadmagnet.py::build_privacy_policy` **хардкодит** строку «6.3. Трансграничная передача ПДн не осуществляется». На текущем live-бэкенде (AI Gateway / обычный cloud-ai) это **ложно** — Timeweb официально подтвердил (2026-06-28): Gateway и обычные cloud-ai агенты трансграничны (посредник к внешним провайдерам, данные могут покидать РФ, ZDR нельзя). Нетрансгранична только **приватная** услуга (инференс на мощностях Timeweb в РФ).

Владелец переходит на **приватную Gemma-4 (РФ, Москва)** — по сигналу Timeweb это делает 6.3 истинной, — но модель **ещё не в тестировании** (не live). Значит: захардкодить «нет трансграна» сейчас = опубликовать ложь на переходе.

Дополнительно Task 1.2 требует **явно раскрыть уход ПДн во внешний ИИ** (checklist C.4 — прозрачность автоматизированной обработки). Сейчас Политика/Согласие про ИИ-обработку молчат.

**Ключевое ограничение резолва:** `_AI_BACKENDS = ("cloud_ai", "gateway")`; `cloud_ai` — это **и** обычный (трансгранично), **и** приватный РФ-agent. По backend+agent_id код **не отличает** трансгран от РФ → нужен **явный сигнал-истина** от платформы.

**3 места генерации** (ни одно не знает про бэкенд):
- `bot-telegram/bot.py::_legal_page` — публичная страница `/legal/{slug}/{privacy|consent}`;
- `admin-panel/app.py:5234-5240` — превью в панели;
- `bot-telegram/db.py:1598-1599` — согласие в воронке (`get_funnel_config`).

## 2. Цель / не-цель

**Цель:**
1. Никогда не публиковать ложную декларацию 6.3; делать её управляемой явным флагом-истиной.
2. Явно раскрыть ИИ-обработку обращений в Политике и Согласии (Task 1.2 / C.4).
3. Дать владельцу правдиво заявить «нет трансграна» — включив флаг в момент реального перехода на РФ-инференс.

**Не-цель (YAGNI):**
- Жёсткий гейт `funnel_enabled` (это был отклонённый подход #2).
- Полная Путь-D-обвязка (уведомление РКН о трансгран, отдельное согласие ст.12) — выбран Путь A.
- Автодетект трансграна из backend/agent_id (невозможен — см. §1).

## 3. Дизайн

### 3.1. Флаг-истина `ai_inference_rf`

- Глобальный ключ в `app_settings` (не tenant-scoped — как читает `get_ai_overrides`). Оба аппа читают одну строку → нет расхождения панель/бот.
- Семантика: `true` = инференс ИИ выполняется в РФ (нетрансгранично) → трансграна нет.
- **Дефолт (ключ отсутствует) = `false`** = «трансгранично» (безопасно: не заявляем «нет трансграна» без подтверждения).
- **Fail-safe:** любая ошибка чтения → `false`. Ложную декларацию не публикуем никогда.
- Читается хелпером `get_ai_inference_rf() -> bool` в `bot-telegram/db.py` и `admin-panel/db.py` (парсинг truthy через существующий `_truthy`/`_opt_bool`).
- Вычисление для генераторов: `transborder = not get_ai_inference_rf()`.

### 3.2. Правки текста (`shared/leadmagnet.py`)

Новый keyword-параметр `transborder: bool = True` (дефолт = безопасный) в `build_privacy_policy` и `build_consent_text`.

**§6.3 — условная (в `build_privacy_policy`):**
- `transborder=False` (РФ подтверждён):
  `«6.3. Трансграничная передача персональных данных не осуществляется.»`
- `transborder=True` (дефолт / трансгранично):
  `«6.3. Для подготовки ответов на обращения оператор использует сторонние автоматизированные сервисы обработки (в том числе системы искусственного интеллекта). При этом персональные данные могут передаваться обработчикам, в том числе за пределы Российской Федерации. Идентифицирующие контакты (номер телефона, адрес электронной почты, ИНН) обезличиваются до передачи в такие сервисы.»`

**§6.5 — новый пункт, всегда обе ветки (раскрытие ИИ, ядро Task 1.2 / C.4):**
`«6.5. Для подготовки ответов на обращения оператор использует автоматизированные системы, в том числе системы искусственного интеллекта. Идентифицирующие контакты (номер телефона, адрес электронной почты, ИНН) обезличиваются до передачи в такие системы.»`

> Пункт «исключительно автоматизированные решения» (ст.16 152-ФЗ) **не заявляем** — отдельная область, оценивает юрист. Раскрываем только факт ИИ-обработки.

**Согласие (`build_consent_text`) — одна строка, всегда**, перед финальной «Нажимая…»:
`«• Для подготовки ответов используются автоматизированные системы, включая ИИ; идентифицирующие контакты обезличиваются.»`
Трансгран-специфику согласие не несёт (детали — в Политике); минимальность согласия сохраняется.

### 3.3. Проводка 3 мест генерации

Каждое место читает флаг и передаёт `transborder`:
- `bot.py::_legal_page` — `rf = await db.get_ai_inference_rf()`; передать в `build_privacy_policy(..., transborder=not rf)` и `build_consent_text(..., transborder=not rf)`.
- `admin-panel/app.py` превью — `rf = await db.get_ai_inference_rf()`; передать в оба билдера.
- `bot-telegram/db.py::get_funnel_config` — `rf` (уже в БД-контексте); передать в `build_consent_text`.

### 3.4. Панель-чекбокс (иначе флаг неоперабелен)

- Раздел «ИИ-агенты» (`/agents`), только `is_platform`.
- Чекбокс «Инференс ИИ выполняется в РФ (нетрансгранично)» → пишет `app_settings['ai_inference_rf']` существующим settings-write-путём (CSRF + `_require_admin`).
- Подсказка: «Включайте только после письменного подтверждения РФ-размещения инференса и логов (см. legal-gate досье). По умолчанию — выкл.: Политика раскрывает возможную трансграничную передачу.»

## 4. Поток данных

`app_settings.ai_inference_rf` → `get_ai_inference_rf()` (бот/панель, fail-safe False) → `transborder = not rf` → `build_privacy_policy/build_consent_text(transborder=…)` → текст 6.3 (условно) + 6.5 (всегда) + строка согласия (всегда).

## 5. Обработка ошибок / fail-safe

- Чтение флага падает / ключа нет → `False` → `transborder=True` → Политика раскрывает возможную трансгран-передачу, «не осуществляется» **не** печатается. Безопасно по умолчанию.
- `build_*` без параметра (легаси-вызов/смоук) → дефолт `transborder=True` → безопасно.

## 6. Тестирование

- **Unit-смоук** `scripts/consent_text_smoke.py` (расширить, без БД):
  - `transborder=True` → 6.3 содержит трансгран-раскрытие, НЕ содержит «не осуществляется»; есть 6.5; согласие содержит ИИ-строку.
  - `transborder=False` → 6.3 = «не осуществляется»; есть 6.5; согласие содержит ИИ-строку.
  - дефолт (без параметра) = ветка `transborder=True`.
- **DB-смоук** (risuy_dev, гард по TEAM_DSN): `get_ai_inference_rf()` — дефолт False; после записи ключа = True; парсинг truthy; fail-safe.
- **3-линзовое адверсариальное ревью (Workflow)** перед коммитом: (1) корректность/проводка await+3 места; (2) комплаенс-формулировки (не переобещать/не недообещать, цитатная гигиена); (3) изоляция флага (глобальный, платформа-only запись, fail-safe).

## 8. Риски / открытые вопросы

- **Формулировку ветки `transborder=True` выверяет юрист** (не переобещать/не недообещать; согласуется с §5 цитатной гигиены досье).
- **Логи Gemma shared vs dedicated** (retention) — отдельный 152-ФЗ-вопрос, **не блокер C** (C = только декларация + раскрытие ИИ; retention — операционный вопрос при переключении).
- **Per-tenant backend override:** флаг глобальный (платформенный). Если тенант переопределит `ai_backend`/`ai_agent_id` на трансгранична при глобальном `rf=true` — теоретическое расхождение. YAGNI сейчас (платформа контролирует бэкенды); зафиксировано как известное ограничение; при появлении per-tenant бэкендов — расширить до per-tenant флага.
- **Прод-DDL не требуется** — `app_settings` KV уже есть, `panel_rw` пишет; ключ `ai_inference_rf` аддитивен.
