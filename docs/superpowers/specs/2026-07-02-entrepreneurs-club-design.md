# Дизайн: «Клуб предпринимателей» — кросс-маркетинговая сеть

**Дата:** 2026-07-02 · **Статус:** спека (перед планом) · **Раздел:** «Клуб» (новый пункт меню панели).

## 1. Контекст и цель

Раздел «База компаний» (ЕГРЮЛ-обогащение) по закону НЕ может использоваться для холодного поиска-и-контакта бизнесов (152-ФЗ + ФЗ-38 ст.18 + ГК ст.1334/1335.1). Вместо обхода запрета — **легальный разворот в opt-in клуб**: бизнесы **сами вступают** в клуб (дают согласие на матчинг и партнёрские предложения), после чего платформа знакомит комплементарных партнёров. Согласие = легальный базис; сеть согласившихся = моат («такого нет ни у кого» — потому что парсеры мертвы и вне закона, а сеть по согласию защитима).

**Цель:** killer-фича для привлечения тенантов — «каждый лид/участник → в кросс-маркетинговую сеть; система сама находит партнёра по цепочке потребления; знакомство по взаимному согласию». Полностью в рамках 152-ФЗ.

**Красная линия (незыблема):** ни одного холодного контакта. Любой контакт — только между согласившимися членами клуба; первый обмен контактами — только по **взаимному** согласию на знакомство.

## 2. Цель / не-цель

**Цель (первый релиз, «всё сразу»):** полный луп (профиль → каталог → матчинг → знакомства) + cross-tenant биржа (Уровень 2) + оба входа (клуб-лидмагнит + промоушен лида).

**Не-цель (вне scope):** in-app чат между членами (первый релиз — обмен контактами при взаимном согласии, чат — потом); платные партнёрские сделки/эквайринг внутри клуба; автоматическая генерация сертификатов/купонов (механики кросс-маркетинга — контентный слой на будущее); аналитика рынка v2 (плотность/белые пятна — отдельная спека).

## 3. Домен (новые сущности)

Архитектура C: отдельный домен «Клуб», переиспользует ЕГРЮЛ (`dadata`/`prospects`) + согласие (`consent_events`) + нотификатор.

**Прод-DDL (новые таблицы, RLS tenant_isolation по `app.tenant_id` — как `consent_events`):**

- **`club_members`** — бизнес в клубе.
  `id uuid pk`, `tenant_id uuid not null → tenants` (владелец = Уровень 1), `lead_id uuid → leads on delete set null` (если промоушен лида; null для клуб-лидмагнита), `inn text` (связь с ЕГРЮЛ-`prospects`, null допустим), `display_name text not null`, `city text`, `okved text`, `status text not null default 'active' check (status in ('active','paused','left'))`, `network_opt_in boolean not null default false` (Уровень 2), `created_at timestamptz default now()`.
- **`club_profiles`** — партнёрский профиль (1:1 к члену).
  `member_id uuid pk → club_members on delete cascade`, `tenant_id uuid not null` (для RLS), `offering text` (что даю — продукт/услуга), `avg_check integer` (ср.чек, опц.), `seeking text` (что ищу), `chain_position text check (chain_position in ('before','after','both'))` (ДО/ПОСЛЕ в цепочке потребления), `okved_seek text` (ОКВЭД искомого партнёра), `description text`.
- **`club_intros`** — знакомства.
  `id uuid pk`, `tenant_id uuid not null` (RLS = инициатор), `from_member uuid → club_members`, `to_member uuid → club_members`, `to_tenant_id uuid` (тенант цели — для cross-tenant intro), `status text not null default 'requested' check (status in ('requested','accepted','declined','cancelled'))`, `message text`, `created_at`, `decided_at timestamptz`.

**Матчи** — вычисляются на лету (без таблицы; YAGNI).

**RLS:** на всех трёх — `tenant_isolation` по `app.tenant_id` (паттерн `consent_events`: `using (tenant_id = nullif(current_setting('app.tenant_id', true), '')::uuid)`). Бот (owner) обходит и фильтрует явно; панель (`panel_rw`) подчиняется. Гранты `panel_rw` — в `db/panel_role.sql` (см. §9).

## 4. Топология / RLS + согласие — 🔴 крух

### 4.1. Уровень 1 (клуб тенанта)
Стандартная `tenant_isolation`. Тенант видит/матчит/знакомит **только своих** членов. Полностью на текущей RLS, без исключений.

### 4.2. Уровень 2 (биржа, cross-tenant)
Член с `network_opt_in=true` виден **другим network-членам** через границы тенантов. Это **сознательное consent-gated исключение** из изоляции:
- Реализуется **отдельной функцией биржи** (`club_network_directory`, `club_network_match`) — путь **owner/платформа**, НЕ обычная tenant-RLS. Функция жёстко фильтрует `network_opt_in=true AND status='active'` и отдаёт **только не-ПДн поля профиля** (display_name/city/okved/offering/seeking/chain — БЕЗ lead_id, БЕЗ контактов).
- Доступ к бирже — только у самих network-членов (тенант, у которого есть ≥1 network-член, ИЛИ платформа). Не-члены биржу не видят.
- Cross-tenant знакомство (`club_intros` с `to_tenant_id` ≠ своего) — только между двумя Уровень-2-членами.
- **Легальный базис cross-tenant видимости = Уровень-2 согласие** («сделать профиль видимым в общей бирже предпринимателей и получать предложения о партнёрстве от участников из других кабинетов»).

⚠️ Это единственное место, где мы намеренно выходим за tenant-изоляцию — реализация ТОЛЬКО через выделенные network-функции с явным фильтром, с максимумом ревью. Обычные `club_*`-запросы остаются строго tenant-scoped.

### 4.3. Согласие (в `consent_events`, append-only 152-ФЗ реестр)
Переиспользуем без изменения CHECK `action ('granted','revoked')`: различаем по `doc_type`:
- `doc_type='club_join'` + `action='granted'` — вступление в клуб тенанта (Уровень 1);
- `doc_type='network_join'` + `action='granted'` — вступление в биржу (Уровень 2); `revoked` — выход;
- `doc_type='intro_accept'` + `action='granted'` — согласие на обмен контактами при знакомстве.
`tenant_id` обязателен; `lead_id` — когда есть; `text_hash` = sha256 текста согласия; `channel`. **Аддитивный DDL:** добавить `consent_events.member_id uuid → club_members(id) on delete set null` (nullable) для трассировки согласий клуб-лидмагнитных членов без лида. RLS `consent_events` уже есть.

## 5. Входы (оба)

- **Промоушен лида:** в карточке лида («Диалоги») кнопка «Пригласить в клуб» → бот шлёт лиду запрос согласия (текст `club_join`) → лид принял → `club_members` (linked `lead_id`) + `consent_events(club_join,granted)`. Опц. допрос на Уровень 2. Reuse воронки/бота/`consent_events`.
- **Клуб-лидмагнит:** новый тип воронки «Вступить в клуб» (по паттерну `FUNNEL_FIELDS`/`build_consent_text`) → регистрация (город, ОКВЭД, «что даю»/«что ищу», согласие Ур.1 + опц. Ур.2) → `club_members` (без `lead_id`) + `club_profiles` + `consent_events(club_join,granted)`.

## 6. Полный луп + семантика матчинга

- **Каталог** (`GET /club`): список членов — Уровень 1 (свои, tenant-scoped) + Уровень 2 (биржа, network-функция); фильтр город/ОКВЭД; ЕГРЮЛ-обогащение по `inn`.
- **Матчинг** (кросс-маркетинг, «цепочка потребления»): партнёр стоит **ДО** тебя (даёт трафик) или **ПОСЛЕ** (даёт upsell); скор по: комплементарность `chain_position` (before↔after) + пересечение `okved`/`okved_seek` + один город. Причина показывается человекочитаемо («он ДО вас в цепочке, тот же город, комплементарный ОКВЭД»).
- **Знакомство** (`POST /club/{id}/intro`): «Предложить знакомство» → цель-член получает уведомление (нотификатор `NOTIFIER_BOT_TOKEN`) → принимает/отклоняет (`club_intros.status`) → при **accepted** обмен контактами (`consent_events(intro_accept,granted)` с обеих сторон зашит в accept). До взаимного «да» контакты не раскрываются.

## 7. Переиспользование
ЕГРЮЛ — `admin-panel/dadata.py` + `prospects` (по ИНН) · Нотификатор — `NOTIFIER_BOT_TOKEN` + `bot-telegram/escalation.py` (паттерн уведомления в бота) · Согласие — `consent_events` (append-only, RLS) · Тексты согласия — паттерн `shared/leadmagnet.py::build_consent_text` (новые тексты для club_join/network_join/intro).

## 8. Комплаенс-слой
Тексты согласия раскрывают явно: Ур.1 — матчинг + партнёрские предложения в клубе тенанта; Ур.2 — cross-tenant видимость профиля в бирже + предложения от участников других кабинетов; intro — обмен контактами при взаимном согласии. Всё пишется append-only в `consent_events` (доказательная база). Красная линия (§1) соблюдена во всех путях: только согласившиеся члены, первый контакт = взаимное согласие.

## 10. Тестирование
- **DB-смоуки (risuy_dev, гард TEAM_DSN):** RLS-изоляция A≠B на `club_members`/`profiles`/`intros`; network-функция биржи (Ур.2) отдаёт cross-tenant ТОЛЬКО `network_opt_in=true` и без ПДн-полей; согласия пишутся в `consent_events` с верными doc_type; intro-флоу (requested→accepted→обмен) + взаимность.
- **Unit/render-смоуки:** тексты согласия (club_join/network_join/intro) через builder; render `/club` (каталог/матчинг/пустые состояния; платформа-под-клиента vs тенант); матчинг-скоринг (before↔after + okved + город) на фикстурах.
- **3-линзовое адверсариальное ревью (Workflow) на каждую пачку:** correctness / **изоляция (RLS + cross-tenant биржа — главный фокус)** / комплаенс (согласия + красная линия). Смоук перед коммитом.

## 11. Порядок сборки (внутри «всё сразу»)
1. Прод-DDL `club_*` + `consent_events.member_id` + panel_role (risuy_dev → прод по «да»).
2. Домен + db-хелперы (`club_member_*`, `club_profile_*`, согласия) + тексты согласия.
3. Входы: клуб-лидмагнит (воронка) + промоушен лида (кнопка в карточке лида).
4. Каталог `/club` (Уровень 1) + ЕГРЮЛ-обогащение.
5. Матчинг (цепочка потребления) + причины.
6. Знакомства (intro-флоу + нотификатор + взаимное согласие).
7. **Биржа Уровень 2** (network-функции cross-tenant) — отдельные, максимально-верифицируемые задачи (крух §4.2).

## 12. Риски / открытые вопросы
- **🔴 Cross-tenant биржа (§4.2)** — единственный выход за tenant-изоляцию; риск межтенантной утечки при ошибке фильтра. Митигация: выделенные network-функции + only-non-PII + only-network-members + max ревью изоляции + db-смоук утечки.
- **Юрист выверяет тексты согласия** (club_join/network_join/intro) — формулировки согласия на матчинг/сеть/обмен контактами (152-ФЗ ст.9 + ФЗ-38: согласие на получение партнёрских предложений). Не боевое включение тенантам до юр-подтверждения (как pre-launch gate DaData).
- **Матчинг-качество** — ОКВЭД+город+цепочка это v1-эвристика; тюнинг скоринга — после первых данных.
- **Нотификатор** — сейчас тестовый `@ConversionConsultant_bot`; для боевого клуба — брендовый бот (как отмечено в хендоффе).
- **Связь члена с каналом лида** (куда слать intro-уведомление, если член пришёл через клуб-лидмагнит) — уточнить канал доставки на этапе плана.
