## 1. Контекст и цель

risuy — reseller-платформа «ИИ-Агент Про» (бот aiogram + admin-panel FastAPI/Jinja2 + Postgres/asyncpg/pgvector, мультитенантность с RLS). Воронка лидов — **B2C**: `/start → согласие → имя → телефон → гейт подписки → выдача лид-магнита`; таблица `leads` хранит `name`, `phone`, `phone_hash`, `consent`, `status` — **ни ИНН, ни названия компании, ни ОКВЭД у лида нет**.

**Цель v1:** дать тенанту/оператору возможность по **ИНН или названию** подтянуть карточку юрлица/ИП из ЕГРЮЛ/ЕГРИП (через провайдера DaData, модель per-lookup), сохранить её в свою базу компаний и/или привязать к уже-согласившемуся лиду. Модель — **точечные запросы**, храним только то, что тенант фактически запросил/привязал, **без телефонов на выгрузку и без масс-экспорта**.

Это первая из двух спек модуля. **v2** (отдельной спекой) — обезличенная агрегатная аналитика рынка (плотность/белые пятна по городу+ОКВЭД из открытых данных ФНС/реестра МСП).

## 2. Область v1 (scope)

**Входит:**
- Прод-DDL: новая tenant-scoped таблица `prospects` (карточки компаний, per-lookup).
- Провайдер-адаптер `dadata.py` (метод find-party по ИНН/ОГРН + suggest party как живое автодополнение по названию), с graceful-degrade и quota-guard.
- Раздел панели «База компаний» (оба контура — тенант и платформа-под-клиента, гейт `active_tenant_id`): поиск → карточка → «Сохранить» / «Привязать к лиду»; список сохранённых.
- Блок «Компания» в карточке лида (раздел диалогов): «Привязать компанию» → показ привязанного prospect.
- Комплаенс-гейты в коде (разделение ЮЛ/ИП, вырезание телефонов/email/закрытых категорий, правовое основание, ретеншн/удаление).

**НЕ входит (defer / YAGNI):**
- Агрегатная аналитика рынка (v2, отдельная спека).
- pgvector семантический дедуп/«похожие компании» (в v1 идентичность — точный ключ ИНН).
- Метеринг/биллинг запросов DaData тенантам (пока платформа поглощает; pre-PMF).
- Массовый импорт/ingest, bulk-выгрузка.
- Функция рассылки «по карточке компании» (её в v1 нет вовсе; любая будущая рассылка — только по лиду с доказуемым opt-in).
- `founders[]` (учредители) — доп. ПДн-поверхность, отложено.

## 4. Источник данных A — DaData find-party (проверено)

DaData (ООО «Дейта Кью», ИНН 7721581040), работа по публичной оферте (ред. 20.04.2026), договор не заключается.

**Тарифы (dadata.ru/pricing, подтверждено адверсариально):** бесплатный — «как Лёгкий, но 10 000 запросов/сутки» + логотип DaData в подсказках; **«Лёгкий» — 14 000 ₽/год, 50 000/сутки** (выбран); «Расширенный» — 28 000 ₽/год, 100 000/сутки; «Максимальный» — 56 000 ₽/год, от 200 000/сутки. Подписка минимум на год (помесячно нельзя). ≈**1 170 ₽/мес** в пересчёте; при 1 тест-тенанте лимит практически неисчерпаем.

**Два метода (критично для дизайна):**
- **find-party (findById, «Организация по ИНН или ОГРН»)** — точный поиск, отдаёт полный набор реквизитов **И телефоны `data.phones[]` (~60% действующих ЮЛ) + email `data.emails[]` (~50%)**. Используем ТОЛЬКО для обогащения по ИНН/ОГРН.
- **suggest party («Подсказки по организациям»)** — нечёткий поиск по названию; телефоны/email **всегда null**; учредителей нет. Используем как **server-round-trip** поиск по названию (оператор вводит запрос → POST → сервер отдаёт список подсказок → оператор выбирает нужную и подтягивает по ИНН через find-party). **Без клиентского JS/typeahead** — канон risuy server-rendered/CSP-safe (прецедент онбординга). Live-typeahead откладываем.

⚠️ **Оферта 4.2.1 запрещает автоматическую обработку через «Подсказки» (suggest).** Поэтому автоматическое/пакетное обогащение — только через **find-party**; suggest — исключительно живой поиск оператора по названию (ручной ввод, ручной выбор), не пакетный.

**Условия хранения/использования:** оферта (раздел 4.2 — исчерпывающий перечень запретов) НЕ содержит запрета на коммерческое использование и на хранение результатов; раздел 8.1 закрепляет исключительные права на сами справочники (ограничивает перепродажу/публикацию/копирование базы, но не использование результатов конкретных запросов). Прямой **разрешающей** формулировки в оферте нет → см. pre-launch gate (письменное подтверждение поддержки DaData).

**Технические лимиты API:** ≤30 запросов/с, ≤60 новых соединений/мин с одного IP, длина query ≤300 символов. Для операторского per-lookup не критично; закладываем простой quota-guard.

**Запасной провайдер:** api-fns.ru/DaMIA (чистый per-lookup REST, от 0,1 ₽/запрос или пакеты 10–40 тыс ₽/год). Провайдер делаем **сменяемым за общим интерфейсом** (см. §6).

## 5. Модель данных — таблица `prospects`

Канон risuy (см. `db/schema_rls_leads_messages.sql`, `db/migrate_consent_events.sql`): tenant-scoped таблица получает `tenant_id not null → tenants(id) cascade`, RLS-политику `tenant_isolation` (**ENABLE, не FORCE**; owner `gen_user` обходит) через `nullif(current_setting('app.tenant_id',true),'')::uuid`, гранты `panel_rw` явные, **без DELETE** (деактивация статусом/флагом).

Таблицу пишет **ПАНЕЛЬ** (операторское действие), не бот. Владелец таблицы — `gen_user`, миграция применяется owner-DSN.

**Файл миграции:** `db/migrate_prospects.sql` (идемпотентно, `if not exists`).

```sql
create table if not exists prospects (
    id             uuid primary key default gen_random_uuid(),
    tenant_id      uuid not null references tenants(id) on delete cascade,

    -- идентичность (точный ключ)
    inn            text not null,
    kpp            text,
    ogrn           text,
    subject_type   text not null default 'legal'
                   check (subject_type in ('legal','individual')),  -- ЮЛ vs ИП

    -- реквизиты ЕГРЮЛ/ЕГРИП (для ЮЛ — не ПДн; для ИП — режим 152-ФЗ)
    name_short     text,
    name_full      text,
    opf            text,
    okved          text,           -- основной код
    okved_name     text,
    okveds         jsonb,          -- доп. коды ОКВЭД
    address        text,           -- юр.адрес ЮЛ; для ИП — ТОЛЬКО город (адрес места жительства НЕ храним)
    region         text,
    city           text,
    status         text,           -- ACTIVE|LIQUIDATING|LIQUIDATED|BANKRUPT|REORGANIZING
    registration_date date,
    liquidation_date  date,
    management     jsonb,          -- руководитель ЮЛ (внутри — ФИО физлица = ПДн; НЕ для рекламы, маскировать в LLM)

    -- связь с воронкой
    lead_id        uuid references leads(id) on delete set null,  -- переживает обезличивание лида (как consent_events)

    -- кэш/аудит/ретеншн
    source         text not null default 'dadata',   -- dadata|api-fns|manual
    raw            jsonb,          -- САНИТИЗИРОВАННЫЙ ответ (без phones/emails/закрытых категорий), для TTL/refresh
    fetched_at     timestamptz,    -- когда получены данные (TTL-рефреш)
    archived       boolean not null default false,    -- soft-архив вместо hard-delete
    created_by     text,           -- оператор (username)
    created_at     timestamptz not null default now(),
    updated_at     timestamptz not null default now(),

    unique (tenant_id, inn)        -- дедуп по ИНН в пределах тенанта (exact, без pgvector в v1)
);

create index if not exists prospects_tenant_lead_idx  on prospects (tenant_id, lead_id);
create index if not exists prospects_tenant_city_okved_idx on prospects (tenant_id, city, okved);  -- задел на v2

alter table prospects enable row level security;
do $$ begin
    if not exists (select 1 from pg_policies where tablename='prospects' and policyname='tenant_isolation') then
        create policy tenant_isolation on prospects
            for all
            using (tenant_id = nullif(current_setting('app.tenant_id', true), '')::uuid)
            with check (tenant_id = nullif(current_setting('app.tenant_id', true), '')::uuid);
    end if;
end $$;

do $$ begin
    if exists (select 1 from pg_roles where rolname='panel_rw') then
        grant select, insert, update on prospects to panel_rw;  -- без delete (канон: archived-флаг)
    end if;
end $$;
```

**Схемно НЕ храним и не заводим полей под:** телефоны/email компании; паспортные реквизиты; дату/место рождения физлиц; адрес места жительства ИП. Эти категории вырезаются на уровне провайдер-адаптера (§6) ДО записи (defense-in-depth: их нет в схеме И их дропает клиент).

Грант зеркалить в `db/panel_role.sql` (канон).

## 6. Компоненты (код, admin-panel)

### 6.1 `admin-panel/dadata.py` — провайдер-адаптер
Прецедент — `admin-panel/mailer.py` (`is_configured()` + graceful-degrade). Async-клиент (httpx/aiohttp) за **общим интерфейсом** `EnrichmentProvider` (чтобы api-fns был drop-in заменой):
- `is_configured() -> bool` — заданы ли `DADATA_API_KEY` + `DADATA_SECRET_KEY`; не заданы → раздел показывает «источник не подключён», код не падает.
- `find_party(inn_or_ogrn) -> ProspectCard | None` — find-party (findById). **Санитизирует ответ**: удаляет `phones`, `emails`, любые закрытые категории (паспорт/ДР/адрес места жительства ИП); нормализует в поля таблицы; определяет `subject_type` (LEGAL/INDIVIDUAL).
- `suggest_party(query) -> list[Suggestion]` — только для живого автодополнения формы (по названию); телефоны и так null.
- `quota_guard()` — глобальный суточный счётчик запросов (ключ в `app_settings`), алерт при 80% лимита, жёсткий стоп на лимите; учитывает техлимиты API. Мягкий per-tenant кап — ключ в `tenant_settings` (опц., но заложить).
- Таймаут + ретрай; ошибки → задачные RU-сообщения (паттерн онбординга/`_integrations_err_text`).

### 6.2 `admin-panel/config.py` — конфиг
Добавить (паттерн `_req`/`_opt_int`/`_opt_bool`): `DADATA_API_KEY`, `DADATA_SECRET_KEY` (опц. — degrade), `DADATA_DAILY_LIMIT` (дефолт 50000), `DADATA_TENANT_DAILY_SOFT_CAP` (опц.), `ENRICHMENT_PROVIDER` (дефолт `dadata`).

### 6.3 `admin-panel/db.py` — хелперы (panel_rw, tenant-scoped через RLS)
`prospect_upsert(...)` (on conflict (tenant_id, inn) do update, обновляет `fetched_at`), `prospect_link_lead(prospect_id, lead_id)`, `prospect_list(tenant_id, include_archived=false)`, `prospect_get(id)`, `prospect_archive(id)`, `prospect_for_lead(lead_id)`. Все — с in-query tenant-backstop (defense-in-depth к RLS, паттерн `kb_list_documents`).

### 6.4 Маршруты + шаблоны
- `GET /companies` — раздел «База компаний» (гейт `active_tenant_id`, A1-паттерн как `/knowledge`): форма поиска (ИНН/название), список сохранённых prospects, empty-state, help_card, кнопка поддержки.
- `POST /companies/lookup` — find-party по ИНН (CSRF, `_ip`/`_ua`, аудит).
- `POST /companies/search` — поиск по названию через suggest party, **server-rendered список подсказок** (без JS-typeahead, CSP-safe), rate-limited; оператор выбирает → find-party по ИНН.
- `POST /companies/save` — upsert prospect.
- `POST /companies/{id}/link-lead` / `POST /companies/{id}/archive`.
- В `_render_dialogs` (карточка лида): блок «Компания» → «Привязать компанию» (поиск → link) → показ привязанного prospect.
- Шаблоны: `companies.html`, `_company_card.html` (партиал, переиспользуется в диалогах); правки `dialogs`-шаблона.

## 7. Поток данных

**Обогащение лида:** оператор открывает лид (`/dialogs`) → «Привязать компанию» → вводит ИНН (или ищет по названию через suggest) → find-party → санитизированная карточка → `prospect_upsert` + `prospect_link_lead(prospect_id, lead_id)` → карточка показывается в блоке «Компания» лида.

**Карточка по запросу:** «База компаний» → поиск по ИНН/названию → find-party → карточка → «Сохранить в базу» (upsert без lead_id) и/или «Привязать к лиду».

Обе точки входа = одна операция: find-party → санитизация → upsert prospect → опц. link. Дедуп по `unique (tenant_id, inn)`.

## 8. Комплаенс-гейты (проверенная юр-рамка)

Подробные проверенные факты с нормами/источниками — в §12. Здесь — инварианты кода.

1. **Разделение по типу субъекта.** `subject_type='legal'` — данные ЮЛ **вне режима 152-ФЗ** (ЮЛ не субъект ПДн, 152-ФЗ ст.3 п.1), согласие не требуется; но `management.name` (ФИО директора-физлица) = ПДн физлица. `subject_type='individual'` — карточка ИП целиком в режиме 152-ФЗ.
2. **Правовое основание.** ЮЛ — 152-ФЗ не применяется. ИП — обработка без согласия на основании **п.11 ч.1 ст.6 152-ФЗ** (данные подлежат обязательному раскрытию по ФЗ-129). НЕ ссылаться на отменённый п.10 ч.1 ст.6 и НЕ на «общедоступность» как самостоятельное основание.
3. **Гейт закрытых категорий.** НЕ тянем и НЕ храним: реквизиты документа, удостоверяющего личность; дату/место рождения; адрес места жительства ИП/физлица. Реализация: этих полей нет в схеме + провайдер-адаптер дропает их до записи.
4. **Гейт телефонов/рекламы.** find-party отдаёт `phones`/`emails` — **вырезаем на клиенте, не храним**. Наличие телефона в ЕГРЮЛ ≠ согласие на рекламу (ФЗ-38 ст.18). В v1 функции рассылки по prospect НЕТ; любая будущая рассылка (звонок/SMS/email) — только по лиду с доказуемым предварительным opt-in именно на рекламу (лог согласия: timestamp, идентификация адресата, содержание, канал).
5. **Ретеншн/удаление.** TTL карточки (данные ЕГРЮЛ меняются) — рекомендация 30–90 дней + фоновый refresh перед показом (`fetched_at`). По требованию субъекта (ИП/физлица) — исключение его данных из карточек (ст.8 ч.2 152-ФЗ); при удалении тенанта — cascade. При отзыве согласия привязанного лида — пометка «нельзя контактировать» и остановка любой коммуникации.
6. **Уведомление субъекта (ст.18 ч.3 152-ФЗ)** для карточек ИП: подготовить шаблон уведомления (6 блоков: оператор+адрес; цель+правовое основание; перечень ПДн; предполагаемые пользователи; права субъекта; источник=ЕГРИП). Применимость освобождения по ч.4 ст.18 — оценить с юристом (см. pre-launch gate), не считать снятым автоматически.
7. **Модель извлечения — строго per-lookup, без bulk** (bulk = высокий риск ст.1334 ГК). ⚠️ **per-lookup НЕ абсолютный щит:** систематическое извлечение несущественных частей запрещено ст.1335.1 п.3 ГК, если неоднократное + противоречит нормальному использованию + ущемляет интересы изготовителя. Меры: rate-limit, quota-guard, не допускать планомерного обхода реестра; не позиционировать per-lookup как гарантию.
8. **PII в LLM.** ФИО/ИНН ИП и ФИО руководителей/учредителей ЮЛ — ПДн; при передаче карточки в LLM — только через KikuAI Masker (fail-closed), общедоступность из-под 152-ФЗ не выводит.
9. **UI-текст.** Раздел проговаривает: данные = аналитика/обогащение, **не** «список для обзвона»; первый контакт с бизнесом — только через opt-in воронку.

## 9. Обработка ошибок и деградация

- DaData не настроен (`is_configured()==false`) → раздел рендерится с плашкой «источник не подключён», действия недоступны, код не падает (паттерн fail-safe mailer).
- Лимит/квота исчерпаны → вежливый отказ, аудит; поиск временно недоступен.
- ИНН не найден / провайдер вернул ошибку/таймаут → задачное RU-сообщение, без стектрейса.
- Провайдер вернул закрытое поле → дропается санитайзером (не доходит до записи).

## 13. Остаточные риски

- Систематический per-lookup может быть переквалифицирован как воссоздание существенной части БД (ст.1335.1 п.3 ГК) — зависит от объёма/регулярности; практика не изучена.
- Правовой статус ЕГРЮЛ как охраняемой БД и правообладатель (ФНС/государство) — требует анализа практики.
- Применимость исключений ч.4 ст.18 152-ФЗ к выгрузке ИП из ЕГРИП не определена — риск нарушения ст.18 ч.3.
- Возможная обязанность уведомления РКН и включения в реестр операторов при систематическом хранении карточек ИП.
- Лицензионные условия канала (DaData/api-fns/ФНС) могут ограничивать хранение/переиспользование сверх закона.
- Пересечение с ЗДР-контуром: ФИО/ИНН ИП и ФИО руководителей — ПДн; без маскировки в LLM — риск 152-ФЗ.
- Штрафной риск рекламы без доказанного opt-in высок (300k–1M ₽ за факт).
- DaData: прямой разрешающей формулировки на хранение в оферте нет (вывод из отсутствия запрета); цены индексируются.
