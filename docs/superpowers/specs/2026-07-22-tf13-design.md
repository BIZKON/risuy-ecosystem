## T-1F-3a — закрыть Important #1 (in-panel), БЕЗ DDL

**Select-time** `subscription_select` (app.py:3683): `pc=compute_plan_change(...)`. Дописать в metadata не-ПДн снимок:
`pc_dir`(direction), `pc_pool_mode`(add|overwrite), `pc_pool_amt`(str µRUB), `pc_period`(keep|extend|fresh),
`pc_old`(old_plan_code|''), `pc_ps`(current_period_start ISO — ЯКОРЬ периода). Итог ~11 ключей < 16.

**Webhook** (app.py платформ-ветка): собрать снимок из `meta`, прокинуть в `activate_subscription_from_payment(..., plan_change_snapshot=snap)`.

**`activate_subscription_from_payment`** — новый параметр `plan_change_snapshot: dict|None`. Под FOR UPDATE
живой подписки решает `_resolve_pool(live, snap, now, plan)` → `(pool_mode, pool_amount, period_mode, needs_reconciliation)`:
- **Полноценные снимки** (`pc_dir ∈ {primary, extend, downgrade}`, pool_mode=overwrite): полная цена = полный пул Inc_new.
  Применяем verbatim (overwrite Inc_new; период fresh/extend). Дрейф не важен (полная цена). `needs_reconciliation=false`.
- **Upgrade-снимок** (`pc_dir='upgrade'`, add, `pc_pool_amt`=оплаченная дельта, keep):
  - live совпал ТОЧНО (`plan_code==pc_old` И `current_period_start==pc_ps`) И период НЕ истёк (`current_period_end>now`):
    add `pc_pool_amt`, keep период. Когерентно. `needs_reconciliation=false`.
  - иначе (дрейф/истёк): add `pc_pool_amt` (СТРОГО оплаченная дельта), период → **fresh** (пул usable, не мёртвый),
    `needs_reconciliation=true`. НИКОГДА не overwrite Inc_new за частичную доплату (закрывает H-3/H-4).
- **Нет снимка** (renewal/legacy): текущий пересчёт (совместимость).

`needs_reconciliation=true` → пишем в аудит `subscription_activated` detail (`needs_reconciliation:true, f, paid`) — T-1F-4 сканит.
Точность: `pc_pool_amt` bigint→str→int (без float). Guard-фолбэк больше НЕ over-grant'ит (ревью H-4 закрыт).

**Смоук `scripts/tf13a_snapshot_smoke.py`** (гард `/risuy_dev`): (1) апгрейд, период НЕ истёк, live совпал → add дельты, keep, no-recon;
(2) апгрейд, период истёк к вебхуку (f_webhook≤0) → add ТОЛЬКО дельты + fresh период (пул usable) + recon-флаг (RED без фикса — было keep→мёртвый пул);
(3) апгрейд, live сменил план (дрейф pc_old≠) → add дельты + fresh + recon, НЕ полный Inc_new (RED: было primary overwrite);
(4) downgrade/extend/primary снимок → overwrite Inc_new когерентно; (5) повтор payment_id идемпотентен.

## T-1F-3b — лендинг-провижининг + B2B, ПРОД-DDL expand

### DDL `db/migrate_subscribe_b2b.sql` (expand-first, идемпотентно)
```sql
-- identity покупателя → tenant-scoped RLS-таблица (НЕ колонки tenants: 152-ФЗ, tenants без RLS)
create table if not exists tenant_billing_identity (
  tenant_id   uuid primary key references tenants(id) on delete cascade,
  buyer_inn   text not null,
  buyer_ogrnip text,
  buyer_subject_type text not null check (buyer_subject_type in ('legal','individual')),
  updated_at  timestamptz not null default now()
);
alter table tenant_billing_identity enable row level security;
do $$ begin
  if not exists (select 1 from pg_policies where tablename='tenant_billing_identity' and policyname='tenant_isolation') then
    create policy tenant_isolation on tenant_billing_identity for all
      using (tenant_id = nullif(current_setting('app.tenant_id', true),'')::uuid)
      with check (tenant_id = nullif(current_setting('app.tenant_id', true),'')::uuid);
  end if;
end $$;

-- признак «пароль задан пользователем» (гейт claim; закрывает M-9)
alter table admin_users add column if not exists password_set boolean not null default false;

-- серверная pre-tenant покупка (ПДн НЕ в metadata ЮKassa; TTL-purge)
create table if not exists pending_service_purchase (
  id            uuid primary key default gen_random_uuid(),  -- purchase_ref (не-ПДн)
  email         text not null,
  buyer_inn     text not null,
  buyer_ogrnip  text,
  buyer_subject_type text not null check (buyer_subject_type in ('legal','individual')),
  is_entrepreneur boolean not null,
  plan_code     text not null,
  offer_version text,
  offer_text_hash text,
  agree_pdn     boolean not null,
  consent_at    timestamptz not null,
  idempotence_key text not null unique,     -- детерминированный, == ЮKassa idempotence_key
  yookassa_payment_id text,
  status        text not null default 'pending' check (status in ('pending','claimed','failed','expired')),
  tenant_id     uuid,
  created_at    timestamptz not null default now(),
  claimed_at    timestamptz
);
create index if not exists pending_service_purchase_ykid_idx on pending_service_purchase (yookassa_payment_id);
create index if not exists pending_service_purchase_reuse_idx on pending_service_purchase (email, plan_code, status, created_at desc);
```
**НЕ трогаем:** `payments` (согласие идёт в `consent_events`), `tenants` (identity в отдельной RLS-таблице), `subscriptions`.
**Гранты `db/panel_role.sql`:** guarded to_regclass do-блоки (закрывает M-13, mass-revoke) — `tenant_billing_identity` select/insert/update; `pending_service_purchase` select/insert/update; `admin_users.password_set` наследует табличный грант. RLS-таблице — как tenant_settings.

### ИНН/ОГРНИП checksum helper `admin-panel/inn_checksum.py`
`valid_inn_or_ogrn(v)` → (kind ∈ {inn10,inn12,ogrn13,ogrnip15}, subject_type ∈ {legal,individual}) | None.
- ИНН-10 (ЮЛ→legal): веса [2,4,10,3,5,9,4,6,8], (Σ mod 11) mod 10 == d10.
- ИНН-12 (individual): n11 веса [7,2,4,10,3,5,9,4,6,8], n12 [3,7,2,4,10,3,5,9,4,6,8], оба mod 11 mod 10.
- ОГРН-13 (legal): (первые 12 как int) mod 11, последняя цифра результата == d13.
- ОГРНИП-15 (individual): (первые 14 как int) mod 13, последняя цифра == d15.
⚠️ subject_type — техническая маршрутизация по длине; ПРАВОВОЕ основание B2B = чек-бокс `agree_entrepreneur` (M-20).

### Вебхук — ветка `kind=='service_landing'` + `purchase_ref`
`db.provision_service_landing(purchase_ref, payment_id, amount_micro, card_last4, offer_version, offer_text_hash)` — **пошагово идемпотентно** (НЕ одна транзакция; H-5/H-6):
1. Держим `pg_advisory_lock(hashtext('svc:'+lower(email)))` на выделенном соединении весь провижининг (сериализация конкурентных вебхуков по email); try/finally unlock.
2. `pending` по id=purchase_ref FOR UPDATE. `claimed`→no-op (идемпотентно). НЕТ строки при валидном purchase_ref → **транзиент** (raise → processed_ok=False → ЮKassa переретраит; M-14).
3. `find_identity('email', email)`:
   - НЕТ → `create_client_account(provider='email', external_id=email, password_hash=random, verified=false)` в try/except UniqueViolation→повторный find (H-5). is_new=true. Записать `tenant_billing_identity`.
   - ЕСТЬ → взять tenant_id/username. Если tenant suspended/canceled → **реактивировать** status→active (owner). Обновить `tenant_billing_identity` (upsert).
4. `activate_subscription_from_payment(tenant_id, plan_code, payment_id, amount_micro, SERVICE_PLAN_PERIOD_DAYS, receipt_email=email, plan_change_snapshot={pc_dir:'primary', pool_mode:'overwrite', pc_period:'fresh', pc_pool_amt:Inc_new})` — **лендинг ВСЕГДА полный пул + свежий период** (закрывает CRIT-1: полная цена = полный пул, реюз НЕ прорачивается).
5. Согласие → `consent_events` (append-only): doc_type='offer', doc_version/offer_version, text_hash=offer_text_hash, action='granted', channel='web', occurred_at=consent_at (+ отдельная строка doc_type='pdn' если agree_pdn). tenant_id уже есть.
6. `pending` status='claimed', tenant_id, claimed_at.
7. is_new ИЛИ `password_set=false` → `create_reset_token(username, ttl=ACCOUNT_CLAIM_TTL_MIN)` + `mailer.send_account_claim(email, url, ttl)`. Активированный (`password_set=true`) → письмо НЕ шлём (owner). Per-email потолок claim (реюз-гейт ограничивает).

**Частичный сбой:** каждый шаг идемпотентен по своему ключу (identity — find_identity/UniqueViolation; activate — payments yk_payment_id on-conflict; pending — status). Ретрай вебхука восстанавливает (M-6).
