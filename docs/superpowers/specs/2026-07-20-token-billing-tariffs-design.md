## 0. Принятые решения (D1–D5)
- **D1. Юр-модель — ГИБРИД:** тариф = **абонплата за доступ (ст. 429.4 ГК)**, месячный пул токенов = включённый лимит использования, неиспользованное **законно сгорает**; **кошелёк-пополнение = возвратный аванс** (ст. 782 ГК / ЗПП соблюдаем).
- **D2. Гейт покупателя:** онлайн-оплата собирает **ИНН/ОГРНИП + чек-бокс «предпринимательская деятельность»** (держим B2B, снижаем риск ЗПП физлиц; подпирает сгорание-абонплату D1).
- **D3. Автосписания:** **«без автосписаний»** — снять `save_payment_method=True`, `SERVICE_RENEWAL_ENABLED=0`; лендинг-claim честен.
- **D4. Целевая маржа LLM:** **76,5 %** через per-resource прайс (курс ×4,26 в токене; не единый ×3).
- **D5. Фискализация:** **двухчековая схема** (аванс при топапе + зачёт при списании), `SERVICE_RECEIPT_ENABLED=1` до первого платежа по новой модели.

## 1. Цель
Перейти на **единую токен-модель** (образ ElevenLabs): один пул токенов на всё, пополняемый ₽-кошелёк на сверх-пул, всё включено. Мотив: конверсия (одна покупка), прозрачная маржа (в ₽ и %), закрыть неучтённые утечки расхода, расширяемость под новые опции.

## 2. Модель
- **Единица — токен.** Единый пул на всё: чат + опции.
- **Юр-модель (D1):** тариф — **абонплата** (пул месяца сгорает — законно); кошелёк-пополнение — **возвратный аванс**.
- **Позиционирование (обязательно):** «токены» подаём как **включённый лимит использования подписки**, НЕ «вы купили N токенов» (иначе сгорание оспоримо как удержание аванса). Формулировки — в оферте/лендинге.
- **Двойной прайс:** X токенов (из пула) / Y ₽ (из кошелька сверх пула). Курс **1 токен = 0,0015 ₽**.
- **Авто-фолбэк:** сначала пул тарифа, затем кошелёк-аванс. **Хард-стоп** при обоих балансах ≤ 0 (`ai_wallet_blocked`; дефолт-тенант Школы исключён).
- **UX:** два бакета показываем единым «запасом» + разбивка по разделам.

## 5. Прайс за действие + маржа (per-resource, D4/F11)
**Ценовой инвариант:** курс (продажа токена) хранится ОТДЕЛЬНО; наценка задаётся **per-resource**, не единым множителем плана.
- **LLM-действия:** списываем **фактический расход токенов** × курс 0,0015 ₽ (наценка ×4,26 вшита в курс → маржа 76,5 %). НЕ «себест.÷курс».
- **Не-LLM (DaData/голос):** токен-ставка = себест. × наценка ÷ курс (**округление ВВЕРХ**), НЕ break-even.

| Действие | Себест. ₽ | Токен-ставка | ₽ сверх пула | Наценка | Маржа % |
|---|--:|--:|--:|:--:|--:|
| Чат-ответ (~5000 ток) | 1,76 | 5 000 (факт.) | 7,50 | ×4,26 | 76,5 % |
| 🔜 Маркетинг-пост (~8000 ток) | 2,82 | 8 000 (факт.) | 12,00 | ×4,26 | 76,5 % |
| Проверка контрагента (DaData) | 7,50 | 15 000 | 22,50 | ×3 | 66,7 % |
| Поиск клиентов (DaData) | 7,50 | 15 000 | 22,50 | ×3 | 66,7 % |
| 🔜 Голос-минута (EL) | 9,00 | 12 000 | 18,00 | ×2 | 50 % |
| 🔜 Аренда номера/мес | 420 | — | 420–590 | pass-through | — |

**Маржа на пул:** Эконом/Стартовый (чат+DaData) — 66,7–76,5 %; Профи (+голос) — пол 50 %.
**F1:** «5000 ток/ответ» и доля вход/выход `AI_OUT_TOKENS_SHARE=0,5` — **калибровать по факту** из `usage_ledger` (V4 Pro Thinking жжёт больше); маржа/пулы провизорны до замера; 76,5 % — цель, приёмка на реальных данных.

## 6. Кошелёк / бакеты (D1/F10)
Сейчас один `balance_microrub` = **безлимитный перенос** (`renew` добавляет пул); «сгорание» НЕ реализовано. Нужны **2 бакета:**
| Бакет | Логика | Возврат |
|---|---|---|
| **Пул тарифа** (`included_microrub` + `period_end`) | обновляется помесячно, остаток **сгорает** (абонплата) | нет (абонплата) |
| **Кошелёк-аванс** (`topup_microrub`, бессрочный) | пополнение любой суммой | **да** (аванс, по требованию) |

Порядок списания: **пул → кошелёк**. Хард-стоп при обоих ≤ 0.
| Параметр | Значение |
|---|---|
| Курс | 1 токен = 0,0015 ₽ (хранить отдельно) |
| Мин. пополнение | 100 ₽ (`WALLET_TOPUP_MIN_RUB`) |
| Автосписания (D3) | нет; `save_payment_method` снять, `SERVICE_RENEWAL_ENABLED=0` |
| Возврат аванса | да (кошелёк), по требованию |

## 7. Миграция

**Уже построено, переиспользуем:** `credit_wallets` (µRUB) + `usage_ledger` + `charge_usage()` (идемпотентность/FOR UPDATE/округление вверх); замер токенов (`metering_worker.py` + `ai.py::_capture_gateway_usage`); пополнение `POST /wallet/topup` + вебхук `mark_topup_succeeded`; хард-стоп `ai_wallet_blocked`.

**7.1. Слой курса / per-resource прайс (F11).** Курс token↔₽ отдельно; LLM множитель=1 (наценка в курсе), DaData ×3, голос ×2. Инвариант «курс × наценка ресурса = целевая наценка».
**7.2. Модель 2-бакетов (F10).** Миграция схемы: `included_microrub`+`period_end` / `topup_microrub`; списание пул→кошелёк.
**7.3. Схлопнуть 2 счётчика + процедура разреза (F9).** Депрекейт `billing_mode='per_message'`, планы → `cost_multiplier`; депрекейт `service_invoices.quota/overage` + метрики `count_ai_messages`. **Разрез:** на границе периода — заморозить старый счётчик → финальный overage-счёт СТАРОЙ единицей → перенести/обнулить теневой минус кошелька → старт токен-пула со следующего периода; **идемпотентный маркер + запрет пересекающегося окна** (оба счётчика не работают одновременно).
**7.4. Починить `/service/subscribe` + реконсиляция (F13).** Платёж должен нести `tenant_id` (сейчас `kind='service_landing'` без него → вебхук no-op); + **сверка осиротевших платежей ЮKassa за период** (восстановить `payments` + провижининг); pre-tenant провижининг для лендинг-оплаты.
**7.5. Закрыть утечки.** Тарифицировать в токенах: эмбеддинги+RAG на каждый ответ, индексация БЗ, суммаризация памяти, DaData. Модель без строки в `model_prices` → расход бесплатен: добить прайс.
**7.6. Оферта + лендинг (D1/D2/D3).** Переписать: п.1.6 «Сообщение ИИ»→«токен/лимит», п.4 квоты→пул-абонплата, п.4.4 + лендинг-claim «без автосписаний» (снять рекуррент), добавить B2B-гейт (ИНН/ОГРНИП), возврат аванса кошелька. Новая редакция с датой + уведомление действующих клиентов.

## 8. Фискализация 54-ФЗ (D5/F2)
Сейчас: `SERVICE_RECEIPT_ENABLED` дефолт OFF; топап бьётся `payment_mode=full_payment`/`payment_subject=service`; чека зачёта нет.
**Двухчековая схема:**
- **Топап кошелька** → чек «Аванс» (`payment_mode=advance`, `payment_subject=payment`).
- **Списание из аванса** (расход) → чек «Полный расчёт» с зачётом аванса + **номенклатура из `meta.kind`** (llm/embedding/message/other → человекочитаемо).
- **Оплата тарифа** (абонплата) → чек «услуга/подписка».
`SERVICE_RECEIPT_ENABLED=1` в проде ДО первого платежа по новой модели; нужен ОФД/ЮKassa-доступ владельца.

## 9. Апгрейд / даунгрейд / proration (F12)
Сейчас `activate` вставляет НОВУЮ `active`-строку + полный пул на каждую оплату (старая остаётся active) → риск двойной абонентки; даунгрейд держит завышенный баланс.
**Правило:** смена тарифа = **UPDATE существующей подписки** + деактивация прежней + **пропорция цены** + корректировка пула на разницу. **Инвариант: `list_due_renewals` видит ровно ОДНУ живую подписку на тенанта.**

## 15. Критерии приёмки
- Один пул токенов, один счётчик (старая квота депрекейтнута; двойного списания нет; **shadow-diff old vs new на окне разреза = 0**).
- 2 бакета: пул сгорает помесячно, аванс кошелька возвратен; списание пул→кошелёк.
- Per-resource маржа измерена на реальном `usage_ledger` (LLM 76,5 %); не ниже полов (66,7 % / 50 %).
- Смена тарифа не даёт двойной пул; ровно одна живая подписка на тенанта.
- Фискализация: аванс+зачёт бьются; `SERVICE_RECEIPT_ENABLED=1`.
- Ранее неучтённые расходы (эмбеддинги/DaData) списываются.
- Лендинг == бэкенд == оферта; B2B-гейт на оплате; «без автосписаний» честно.
- Школа при balance<0 не встаёт; прогон на тест-магазине 1379463.
