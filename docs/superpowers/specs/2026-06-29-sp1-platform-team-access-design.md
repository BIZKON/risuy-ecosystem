# Спека — A1: «ИИ-команда клиента» в панели владельца-платформы

**Дата:** 2026-06-29
**Проект:** risuy-ecosystem · `admin-panel`
**Статус:** дизайн одобрен (брейншторм 2026-06-29) → готов к плану реализации
**Связано:** СП-1 «ИИ-команда» (`da5f54e`); roadmap `docs/superpowers/specs/2026-06-29-agent-team-departments-roadmap.md`. Это закрытие UX-хвоста СП-1, **часть A1**. **A2** (ретайр орфана `/my-agent`) — отложена отдельной задачей.

## 1. Проблема и цель

Владелец платформы (@profysales, `is_platform`) **не видит** раздел «ИИ-команда» (`/my-team`): по дизайну СП-1 он tenant-only (в навигации только в non-platform-блоке). Поэтому владелец не может настраивать команду ИИ-агентов клиента из своей панели.

**Цель A1:** дать владельцу управлять командой агентов **любого клиента**, переключившись на него, **переиспользуя существующее**: reseller-switch (`POST /tenants/switch` + `admin_sessions.active_tenant_id`) и CRUD СП-1. **Без БД, без правок резолвера/бота.**

## 2. Текущее состояние (факты из кода)

- `templates/base.html` — nav разделён по `is_platform`. Платформа: «ИИ-агенты» (`/agents`), «Базы знаний», «Интеграции», «Демо-монитор» (верхний блок) + «Клиенты» (`/tenants`), «Команда» (`/team`) внизу. Тенант: «ИИ-команда» (`/my-team`, строка ~105) + «Триггеры». `session.is_platform` уже используется в шаблоне.
- `/my-team` (`app.py:5146`) и POST-экшены `/my-team/save` (5180), `/default` (5219), `/channel` (5234), `/disable` (5256) — только `Depends(require_session)`, **без admin-гейта**; все работают на `session.active_tenant_id`. `session` передаётся в контекст шаблона.
- Reseller-switch: `POST /tenants/switch` (`app.py:5375`) пишет `admin_sessions.active_tenant_id`; `tenants.html` — список клиентов + форма switch + бейдж «активный». `auth.Session` отдаёт `active_tenant_id` / `active_tenant_name`.
- `my_team.html:16-19` — при `has_tenant=False`: «Команда настраивается в кабинете клиента… напишите в поддержку» — ориентировано на тенанта, для платформы неверно.
- Эскалация покрыта по-агентно в `/my-team/save` (`escalation_chat_id` / `escalation_topic_id`).
- `/my-agent` (`app.py:4639`) — орфан (вне nav), правит **легаси** ai-config (`set_tenant_ai_config`, фолбэк резолвера) + tenant-эскалацию (`set_tenant_escalation_config`, читается и в `app.py:5003`) → **A2, отложено** (нельзя редиректить, не убедившись, что легаси+эскалация полностью покрыты командой).

## 3. Дизайн (изменения только в `admin-panel`; код-правки — 2 шаблона)

### 3.1 Навигация — пункт для платформы
`templates/base.html`, в `is_platform`-блок (сразу после «ИИ-агенты»):
`{{ nav_item('my_team', '/my-team', 'ИИ-команда клиента', active) }}`.
Метка **«ИИ-команда клиента»** (не «ИИ-команда») — отличает от платформенного «ИИ-агенты» (глобальные настройки Школы) и сигналит, что раздел работает под выбранного клиента. Тенантский блок без изменений (там остаётся «ИИ-команда»).

### 3.2 Ролевое состояние «клиент не выбран»
`my_team.html`, ветка `{% if not has_tenant %}` — ветвить по `session.is_platform` (уже в контексте, **handler менять не нужно**):
- платформа → «Выбери клиента в разделе „Клиенты"» + ссылка на `/tenants`;
- тенант → текущий текст «…напишите в поддержку».

### 3.3 Бейдж активного клиента
`my_team.html`: при `session.is_platform and has_tenant` — вверху бейдж **«Клиент: {{ session.active_tenant_name }}»** (видно, чью команду правишь; снижает риск правки не того клиента — принцип «всё видно/понятно тенанту-владельцу»). `active_tenant_name` доступен через `session`.

### 3.4 Доступ / RLS (без правок кода — верификация на impl)
`/my-team` GET+POST уже работают на `active_tenant_id` без admin-гейта → правки автоматически применяются к выбранному клиенту. Аудит уже пишется (`actor=session.actor`). **Проверить на impl:** `db.set_active_tenant(session.active_tenant_id)` отрабатывает для запросов платформы (`app.py:202`) → RLS-скоуп на активного клиента. Если нет — это блокер (см. §6).

## 4. Не делаем (YAGNI)
- A2: `/my-agent` (редирект/ретайр) — отдельной задачей.
- Страница-деталь клиента; полный «режим клиента» (весь кабинет тенанта в платформе); правки БД / резолвера / бота.

## 5. Тестирование и ревью (ритм risuy)
- **Рендер-смоук `/my-team`**: платформа без активного клиента → CTA на «Клиенты»; платформа с активным клиентом → команда клиента + бейдж; тенант → без изменений.
- Существующий `scripts/team_agents_db_smoke.py` (CRUD под RLS на `risuy_dev`) подтверждает запись под выбранным тенантом.
- **3-линзовое адверсариальное ревью** перед коммитом: correctness; security (платформа через `/my-team` не выходит за RLS-скоуп `active_tenant`); UX (ясность бейджа/CTA, нет тёзка-путаницы в платформенном меню).
- Смоук зелёный + ревью 0 critical/high → коммит. **Push — по «да» владельца.**

## 6. Риски / нюансы
- Платформа правит команду клиента «как клиент» — намеренно; аудит фиксирует `actor`.
- Тёзка-риск в платформенном меню («ИИ-агенты» vs «ИИ-команда клиента») — снимается явной меткой + бейджем клиента.
- ⚠️ **RLS-блокер:** если `set_active_tenant` не вызывается для платформенных запросов на запись — правки уйдут не в тот скоуп; обязательная проверка на impl (вероятно вызывается глобально, `app.py:202`).
