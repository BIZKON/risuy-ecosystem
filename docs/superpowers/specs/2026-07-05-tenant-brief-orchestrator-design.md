# Дизайн: Бриф-онбординг тенанта + Центр принятия решений + Оркестратор

**Дата:** 2026-07-05
**Статус:** дизайн (спека), реализация не начата
**Автор:** brainstorming-сессия (владелец @profysales)
**Тип:** новая фича, subagent-driven реализация после плана

---

## 1. Контекст и цель

Каждый новый тенант («клиент» reseller-платформы «ИИ-Агент Про») должен пройти
профессиональный **бриф-опрос**, где он выгружает суть своего бизнеса: продукты и
оферы, кто его клиенты, как он продаёт, тон общения, каналы и регулярные анонсы.
Цель брифа — собрать всё, что нужно, чтобы **ИИ-сотрудник** тенанта (работающий в
его Telegram-канале: до-продажи, ответы, анонсы) реально давал результат.

Опрос подаётся как **лендинг под конкретного клиента** — по личной ссылке, с
брендингом под имя клиента. Это первый контакт, который должен показать, что за
сервисом стоит профессиональная команда с точными вопросами, а не «Google-табличка».

Ответы попадают в **Центр принятия решений** внутри платформенной админки. Там
**оркестратор** (LLM-модуль) превращает ответы в готовый **черновик настройки**
системы ботов + человекочитаемые рекомендации. Черновик применяет **человек**
(ваша платформенная команда) — одной кнопкой, посекционно, после ревью.

### Что уже есть в системе (опора реализации)

- **Онбординг тенанта:** welcome-карточка + 5-шаговый чеклист (`admin-panel/onboarding.py`,
  роуты в `admin-panel/app.py`), сигналы из реальных данных.
- **Модель тенанта:** `tenants(id, slug, name, status)`, создаётся сразу `active`
  (`db.create_tenant_admin`).
- **Настройки:** `app_settings` (глобальный KV) и `tenant_settings(tenant_id, key, value)`
  (пер-тенантный, RLS).
- **«Ручки» ИИ-сотрудника, которые будет крутить оркестратор:**
  - персона/промпт: `ai_persona`, `ai_persona_role__<slug>`, `ai_persona_tasks__<slug>`,
    `ai_persona_behavior__<slug>` (до 200k), `ai_persona_knowledge__<slug>`, `persona_registry`;
  - продукты/оферы: таблица `products` (kind, name, caption, price, currency, link);
  - воронка: `funnel_config` (`funnel_enabled`, `welcome_text`, `operator_name/inn/email`,
    `leadmagnet_*`, `gate_*`) в `tenant_settings`;
  - команда агентов: `team_agents` + привязка `agent_for_channel__<source>`;
  - база знаний / RAG: `kb_chunks`, `kb_enabled`;
  - триггеры/анонсы: `tenant_triggers` (стоп-слова, интенты, message_count).
- **Публичный лендинг-паттерн:** `_club_landing` / `_club_landing_html` в
  `bot-telegram/bot.py` (`GET /club/{slug}`, aiohttp, без авторизации, самодостаточный
  HTML + встроенный CSS, `Cache-Control: public, max-age=600`); бот читает данные
  тенанта по слагу через `db.get_legal_doc_data(slug)` без сессии.
- **CSRF/формы в панели:** `_enforce_csrf` (`admin-panel/app.py`), `csrf_token` в
  шаблонах, `db.audit(...)` перед сайд-эффектом (пример `/club/export.csv`).
- **Задел на оркестратор:** флаг `team_agents.is_orchestrator` (в БД, без логики).
- **Обезличивание (152-ФЗ):** слой PII-маскировки уже в системе (обезличивание LIVE).
- **LLM-бэкенд:** cloud-ai / OpenAI-совместимый gateway (сейчас DeepSeek), паттерн
  структурированного JSON-ответа отработан в воркерах (`response_format=json_object`,
  reasoning-headroom, truncation-retry, timeout 420s).

## 2. Зафиксированные решения (развилки брейншторма)

| # | Решение | Выбор |
|---|---|---|
| 1 | Как ответы превращаются в настройку | **Оркестратор → черновик + HumanGate** (человек применяет) |
| 2 | Где бриф и кто применяет | **Managed:** публичный брендированный лендинг → черновик применяет **ваша команда** (`is_platform`) |
| 3 | Кастомизация брифа под клиента | **Умный стандартный бриф + брендинг** (единый опрос с ветвлением, брендирован под имя клиента; вопросы под каждого не пишем) |
| 4 | Где живёт лендинг | **Сплит:** лендинг в **боте** (публичное лицо), оркестратор/применение в **панели** |

**Инвариант архитектуры:** оркестратор — чистая функция без побочных эффектов;
запись в прод-настройки происходит ТОЛЬКО в панели за HumanGate, через те же
db-сеттеры, что и ручные формы.

## 3. Архитектура и поток данных

Три части: **публичное лицо (бот)** · **общий слой (БД + схема брифа)** ·
**внутренний контур (панель)**. Логику диалогов бота не трогаем — добавляем только
публичные роуты лендинга.

```
            ВАША КОМАНДА                         КЛИЕНТ (тенант)
                 │                                     │
   1. «Создать бриф-ссылку» ─────────┐                 │
      (панель, is_platform)          │                 │
                                     ▼                 │
                          [tenant_brief]  token, status=pending
                                     │                 │
                 2. личная ссылка ───┼──────────────►  │
                    /brief/{token}   │                 │
                                     │                 ▼
                                     │      3. проходит опрос (БОТ, публичный
                                     │         GET/POST /brief/{token}, брендинг)
                                     │                 │
                          answers ◄──┼─────────────────┘  status=submitted
                                     ▼
            4. ЦЕНТР РЕШЕНИЙ (панель): видит ответы
                     │
                     ▼  «Собрать черновик»
            5. ОРКЕСТРАТОР (панель): answers + схема брифа
                     │   → обезличивание → LLM (существующий бэкенд)
                     ▼   → proposal JSON + рекомендации   status=proposed
            6. HumanGate: предпросмотр-диф, правка, «Применить» (по секциям/всё)
                     │
                     ▼   пишет в tenant_settings / products / team_agents / app_settings
            7. НАСТРОЙКА ЖИВА   status=applied   (аудит на каждом шаге)
```

**Единый источник истины — `shared/brief_schema.py`:** определение секций, вопросов,
типов, ветвлений, `maps_to`. Читают И бот (рендер формы), И оркестратор (интерпретация
ответов). Изменение вопросов — в одном месте.

**Границы модулей:**
- **Бот** — читает схему, рендерит лендинг, пишет `answers` по токену. Не знает про
  оркестратор.
- **Оркестратор** (`admin-panel/brief_orchestrator.py`) — чистая функция
  `analyze(answers, schema) → proposal`. Не пишет в прод.
- **Панель (применение)** — единственное место записи proposal в реальные настройки,
  под `is_platform`, с аудитом.

## 4. Модель данных

### 4.1 Таблица `tenant_brief`

Одна активная запись на тенанта; история — через версии/`created_at`.

| Поле | Тип | Назначение |
|---|---|---|
| `id` | PK | |
| `tenant_id` | FK→tenants | чей бриф (in-query backstop + RLS) |
| `token` | text UNIQUE | секрет публичной ссылки (~16 байт hex, как `password_reset_tokens`) |
| `status` | text | `pending`→`submitted`→`proposed`→`applied` (+`expired`) |
| `answers` | jsonb | ответы клиента (пишет бот по токену) |
| `proposal` | jsonb | черновик оркестратора |
| `applied` | jsonb | снимок принятого (какие секции применены) |
| `created_by` | text | actor вашей команды |
| `created_at` | timestamptz | |
| `submitted_at` | timestamptz | |
| `proposed_at` | timestamptz | |
| `applied_at` | timestamptz | |
| `expires_at` | timestamptz | срок жизни ссылки (по умолчанию +30 дней) |

**Гранты и доступ:**
- Роль бота: SELECT/UPDATE `answers/status` по токену (узкий путь submit). Резолв
  `tenant_id` сервер-сайд по токену — прецедент `get_legal_doc_data(slug)`.
- Роль панели: полный RW.
- **RLS: БЕЗ tenant-isolation RLS** (решение подтверждено при планировании 2026-07-05).
  Обоснование: `tenant_brief` — кросс-тенантный платформенный артефакт (как таблица
  `tenants`); центр решений показывает брифы всех клиентов сразу, а `panel_rw` под
  tenant-RLS вернул бы только активного тенанта. Доступ гейтится приложением
  (`_require_admin`/`is_platform`) + секретностью `token`. Тенант напрямую таблицу не
  читает (managed-модель). Применение настроек по-прежнему идёт через
  `set_active_tenant(tid)` → RLS-сеттеры (`funnel`/`triggers`/`channels`) остаются
  скоуплены на тенанта.

**DDL — аддитивный**, отдельной миграцией `db/migrate_tenant_brief.sql`. Накат по
явному «да» владельца: asyncpg-applier с гардом имени БД, сначала `risuy_dev`, затем
прод `risuy`.

### 4.2 `shared/brief_schema.py`

Версионированная схема — единый источник для бота и оркестратора.

```python
BRIEF_VERSION = 1
SECTIONS = [
  {"key": "business",  "title": "О бизнесе",          "questions": [...]},
  {"key": "products",  "title": "Продукты и оферы",   "questions": [...]},
  {"key": "audience",  "title": "Клиенты",  "branch_on": "b2b_or_b2c", "questions": [...]},
  {"key": "voice",     "title": "Тон и стиль",        "questions": [...]},
  {"key": "channels",  "title": "Каналы и анонсы",    "questions": [...]},
  {"key": "legal",     "title": "Реквизиты оператора","questions": [...]},
]
```

- Тип вопроса: `text | textarea | choice | multichoice | repeatable`.
- Каждый вопрос: `key`, `type`, `label`, `required`, опц. `options`, опц. `maps_to`
  (подсказка, на какую «ручку» влияет ответ — оркестратор использует как ориентир,
  не жёсткий маппинг).
- Ветвление: `branch_on` в секции + условия показа вопросов (напр. `b2b_or_b2c`
  открывает разные доп-вопросы «Клиентов»).
- Хранение ответов: `{"version": 1, "<section_key>": {"<question_key>": value}}` —
  переживает эволюцию схемы (номер версии в ответах).

**Флагманские формулировки** (сигналят экспертизу):
- Продукты: «Что покупают чаще всего и что приносит больше всего прибыли — это одно
  и то же?»
- Клиенты: «В какой момент клиент понимает, что вы ему нужны? Опишите триггер, а не
  портрет.»
- Тон: «Как ваш лучший продавец отвечает на "дорого"? Дайте пример фразой.»
- Анонсы: «Что важного происходит регулярно, о чём стоит напоминать подписчикам?»

*(Финальный список вопросов уточняется на этапе плана; владелец может добавить
обязательные вопросы.)*

## 5. Бриф-лендинг (публичное лицо, бот)

**Роуты в боте** (aiohttp, публичные, паттерн `_club_landing`):
- `GET /brief/{token}` — рендер опроса. Резолв тенанта по токену; `404`/страница
  «ссылка недействительна» при неизвестном/`expired` токене; при `submitted/applied`
  — «бриф уже получен».
- `POST /brief/{token}/submit` — валидация токена+срока, серверная валидация ответов
  по схеме, запись `answers`, `status=submitted`, аудит (IP/UA, без сессии), редирект
  на «спасибо».

**Рендер:** бот читает `shared/brief_schema.py` → собирает самодостаточную страницу.
HTML-шаблон-файл `bot-telegram/templates/brief.html` с плейсхолдерами; JSON-схема
инжектится, минимальный ванильный JS строит поля и шаги. Не Python-простыня. Чистое
клиентское лицо (как `service-site`), мобайл-фёрст, шаг-за-шагом с прогресс-баром.

**Брендинг под клиента (v1):** заголовок + персональное обращение из `tenants.name`
(«Бриф для {Название}»), акцентный цвет. Лого/кастом-цвет — задел на потом.

**Ветвление:** `b2b_or_b2c` и ниша открывают разные доп-вопросы; логика на клиенте
(показ/скрытие) + проверка на сервере при submit.

**Целостность / анти-абьюз:** токен секретный; проверка `expires_at`; повторный
сабмит запрещён после `submitted`; лимиты длины полей; базовый rate-limit по IP.

**152-ФЗ на форме:** бриф собирает **собственные бизнес-данные тенанта** (он — ваш
клиент, не субъект ПДн в этом потоке). Поэтому:
- короткое уведомление «вы передаёте бизнес-данные оператору {платформа}» + ссылка на
  Политику;
- явная просьба **НЕ вставлять персональные данные третьих лиц** (клиентов): просим
  портрет/сегмент аудитории, а не списки контактов;
- при передаче ответов в LLM-оркестратор — **обезличивание** (defence-in-depth).

## 6. Оркестратор (`admin-panel/brief_orchestrator.py`)

Чистая функция: `analyze(answers, schema) → proposal`. Не пишет в прод. Изолированно
тестируется на фикстурах.

**Процесс:**
1. Берёт `answers` + схему (с `maps_to`-подсказками).
2. Обезличивание ответов → LLM через существующий бэкенд (клиент cloud-ai/gateway;
   модель из конфига; структурированный JSON: `response_format=json_object` +
   reasoning-headroom + truncation-retry + timeout 420s).
3. Возвращает `proposal`:

```json
{
  "settings": {
    "persona":  {"name": "...", "role": "...", "behavior_prompt": "...", "knowledge": "..."},
    "funnel":   {"operator_name": "...", "operator_inn": "...", "operator_email": "...",
                 "welcome_text": "...", "leadmagnet_kind": "...", "leadmagnet_url": "...",
                 "gate_enabled": false},
    "triggers": [{"kind": "stopword|intent|message_count", "value": "..."}],
    "channels": {"tg": "<team_agent_slug>"}
  },
  "products": [{"name": "...", "price": 0, "currency": "RUB", "desc": "...",
                "link": "...", "kind": "lead_magnet|tripwire|main"}],
  "recommendations": [{"title": "...", "why": "...", "section": "..."}],
  "gaps": [{"field": "operator_inn", "question": "Не указан ИНН — нужен для 152-ФЗ"}]
}
```

**Правила безопасности:**
- **Никаких выдумок.** Нет данных (ИНН, цена, реквизиты) → в `gaps`, не выдумывается.
- **Детерминированный фолбэк.** LLM упал/затаймаутил → `proposal` собирается правилами
  из `maps_to` (прямой перенос очевидного: название→оператор/компания, продукты→`products`)
  + рекомендации-заглушки. Всегда есть полезный результат.
- **Никогда не крешит.** Ошибка парсинга/сети → частичный proposal + пометка, статус
  остаётся `submitted`, кнопку можно нажать ещё раз.

**Идемпотентность:** «Собрать черновик» пересобирает `proposal` заново (перезапись),
`status→proposed`.

## 7. Центр принятия решений + применение (HumanGate, панель)

Весь контур — под `is_platform`.

**Центр решений** (новый раздел, напр. `/brief-center`):
- **Список брифов** по всем тенантам: статус, тайминги, CTA; воронка
  `создано→отправлено→применено`.
- **«Создать бриф-ссылку»** для тенанта → строка `tenant_brief` + токен → показывает
  готовую личную ссылку (скопировать/отправить клиенту).
- **Просмотр ответов** (`submitted`) — по секциям, read-only.
- **«Собрать черновик»** → синхронно гоняет оркестратор (состояние «собираем…»,
  щедрый timeout) → `proposed`.

**Ревью и применение (HumanGate):**
- `proposal` — **диф-предпросмотр**: для каждой настройки *текущее значение → предлагаемое*.
  Редактируемо (правка текста персоны/офера до применения).
- `recommendations` и `gaps` — на видном месте.
- Применение **посекционно** (чекбоксы) или «применить всё». По клику:
  - `db.set_active_tenant(tid)` → записи скоуплены на тенанта (RLS);
  - **те же db-сеттеры, что и ручные формы** (persona→`ai_persona_*`,
    funnel→`tenant_settings`, products→`products`, triggers→`tenant_triggers`,
    channels→`agent_for_channel__*`) — единая валидация/лимиты, никакого параллельного
    пути записи;
  - снимок принятого → `tenant_brief.applied`, `status=applied`, аудит на каждую запись;
  - CSRF из коробки (`_enforce_csrf`).

**Безопасность:** доступ и применение — только `is_platform`; ничего не применяется
само; диф честно показывает перезапись недефолтных настроек.

## 8. Ошибки, тесты, метрики, выкатка

**Обработка ошибок:**
- Лендинг (бот): неизвестный/истёкший токен → страница «ссылка недействительна»; уже
  отправлен → «бриф получен»; ошибки валидации → форма с сохранением введённого и
  подсветкой; сбой записи → мягкая ошибка, ответы не теряются.
- Оркестратор: LLM-сбой/таймаут → детерминированный фолбэк, запрос не крешит.
- Применение: каждая секция атомарно; при сбое записи — сообщаем какую, остальные
  фиксируются, `applied` только по факту.

**Тесты** (конвенция — smoke-скрипты; db-часть гоняет КОНТРОЛЛЕР на `risuy_dev`):
- `brief_schema_smoke` — схема валидна: key/type у каждого вопроса, `maps_to`
  существует, ветвления ссылаются на реальные вопросы.
- `brief_orchestrator_smoke` — форма `proposal` валидна на фикстурах; фолбэк при
  заглушённом LLM даёт proposal; **no-fabrication** (нет ИНН → `gap`).
- `brief_landing_smoke` — резолв токена, рендер секций, валидация сабмита, ветки
  «истёк»/«уже отправлен» (aiohttp; где `.venv-smoke` без aiohttp → `py_compile`,
  фактическая регистрация — на деплое).
- `brief_apply_smoke` (db, контроллер на `risuy_dev`) — применение пишет в нужные
  таблицы через те же сеттеры, скоуплено на тенанта, аудит есть, повтор идемпотентен.
- Регрессия: существующие smoke (онбординг/воронка) зелёные.

**Метрики:** аудит-события `brief_created/submitted/proposed/applied` (без ПДн в
detail); воронка в центре решений.

**Перф (метрологические бюджеты):** лендинг — лёгкая самодостаточная страница без
тяжёлого JS → быстро на мобильном; LLM-вызов не на критическом пути пользователя;
GET-лендинг с коротким кэшем.

**Выкатка:** миграция аддитивная, сперва `risuy_dev` → прод по явному «да»; деплой
`git push docs/security-audit:main` → редеплой обоих (**бот тоже трогаем** — новые
роуты); новых env, скорее всего, нет (переиспользуем `BOT_PUBLIC_BASE_URL`); если
появятся — добавить в `readBindingsFromEnv`.

## 9. Вне скоупа v1 (YAGNI)

- Редактор брифа под клиента (выбран «умный стандарт»).
- Загрузка лого / кастом-цвет (только имя + акцент).
- Авто-применение (только HumanGate).
- Показ черновика самому тенанту (managed = применяет ваша команда).
- Аналитические дашборды сверх базовой воронки.
