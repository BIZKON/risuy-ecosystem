**Дата:** 2026-06-28 · **Статус:** дизайн согласован + ПРОЙДЕН spec-review (3 линзы, против реального кода) →
правки внесены, готов к writing-plans → impl.
**Часть плана 152-ФЗ из 3 кусков** (делаем по порядку): **#1 эта выгрузка** → #2 YandexGPT-бэкенд + тумблер →
#3 «Справка о ПД» в кабинете + публичная страница.

> **Журнал spec-review (2026-06-28).** Адверсариальное ревью 3 линзами (152-ФЗ/право · безопасность/утечки ·

> defence-in-depth. Подробности — по разделам ниже.

## Зачем
По Приказу РКН №140 (с 01.09.2025) + ЕИП НСУД: Минцифры может потребовать **обезличить** базу ПДн и передать
в систему. Если у тенанта нет пайплайна обезличивания — требование не выполнить → штраф (ось 1 «способность
обезличить по требованию», не зависит от места инференса). Цель: тенант **одной кнопкой** отдаёт обезличенный
набор своей базы клиентов. Это и есть «спать спокойно» по статье (habr selectel/943422).

## Метод (Приказ №140, Приложение №2 — «введение идентификаторов» / псевдонимизация)
Прямые идентификаторы заменяются на стабильный код субъекта; отдельно — защищённый справочник соответствия.

## Объём v1
**База ЛИДОВ** (`leads`) активного тенанта. Диалоги (`messages`, свободный текст) — ОТДЕЛЬНЫЙ инкремент
(нужен NER). **Свободный `notes` ТАКЖЕ вне v1 обезличенного файла** (та же причина — нет NER, regex не ловит
свободные ФИО/адреса) — см. раздел «Поле notes». YAGNI-граница ради скорости и точности #1.

### Реальные колонки `leads` (проверено по `db/schema.sql` + всем `db/*.sql` ALTER)
`id, created_at, updated_at, messenger, source, name, phone, phone_hash, consent, subscribed, status,
guide_sent_at, follow_up_1_at, follow_up_2_at, follow_up_3_at, tg_user_id, max_user_id, notes, survey`
(база `schema.sql`) **+** `tenant_id` (migrate_tenant_scope) **+** `vk_user_id` (migrate_layer_c_identity)
**+** `web_session_id` (migrate_web_leads) **+** `max_chat_id` (migrate_c3_channel_outbound) **+**
`erase_requested_at` (schema_admin) **+** `bot_paused`, `unsubscribed_at` (schema_panel_ext) **+**
`ai_persona` (schema_persona) **+** `escalated_at` (schema_lead_escalated).
⚠️ Колонки `subject_id` в БД **НЕТ** — это вычисляемый псевдоним (см. «Код субъекта»). Реализация — БЕЗ DDL.

### Классификация колонок (явный whitelist для SELECT, НЕ `SELECT *`)
**Прямые идентификаторы → НЕ попадают в обезличенный `anon.csv`** (только в gated `map.csv`):
`name`, `phone`, `phone_hash`, `tg_user_id`, `vk_user_id`, `max_user_id`, `max_chat_id`, `web_session_id`.

**Служебные → НЕ в CSV вовсе:** `id` (только вход для псевдонима), `tenant_id` (скоупит RLS), `survey` (пусто).

**Свободный текст → НЕ в anon.csv (нет NER):** `notes` → в anon.csv только производный boolean `has_notes`.

**Остаются в `anon.csv`** (не идентифицируют по отдельности, полезны регулятору):
`subject_code` (вычисляемый, первый столбец), `messenger`, `source`, `consent`, `subscribed`, `status`,
`created_at`, `updated_at`, `guide_sent_at`, `follow_up_1_at`, `follow_up_2_at`, `follow_up_3_at`,
`unsubscribed_at`, `erase_requested_at`, `ai_persona` (валидировать против `config.PERSONA_PRESETS`; невалидное →
пусто), `bot_paused`, `escalated_at`, `has_notes`.

> ⚠️ **Квази-идентификаторы.** Набор точных timestamp'ов + `source`/`messenger` на малом тенанте — богатый
> quasi-identifier (косвенная реидентификация при сопоставлении с внешними данными). v1: timestamp'ы оставляем
> точными (НСУД-контур может требовать точность), но в UI/сопроводиловке — дисклеймер: набор для передачи в
> ЗАЩИЩЁННЫЙ контур НСУД, не для публикации. Огрубление до даты/часа — опция следующего инкремента.

## Поле `notes` (правка spec-review, critical)
`shared/pii` маскирует только телефон/email/ИНН regex'ом и НЕ ловит свободные ФИО/адреса (докстринг pii.py
строки 8-11). Передать регулятору файл с пометкой «обезличенный», где в `notes` остался открытый ФИО/адрес =
НЕ обезличивание по ст. 3 п. 9 152-ФЗ + самостоятельный факт раскрытия ПДн. Репозиторий уже фиксирует это
ограничение (`docs/152fz-legal-review-checklist.md`). → **В v1 raw-`notes` в `anon.csv` НЕ включаем** (только
`has_notes` boolean). Сырой `notes` доступен ТОЛЬКО в gated `map.csv` (защищённый справочник, см. ниже). Термин
«обезличено частично» из спеки удалён — промежуточного правового статуса нет.

## Код субъекта (`subject_code`)
Вычисляемый в Python, **НЕ колонка БД**. Детерминированный из `lead.id` (uuid v4, ~122 бита энтропии →
необратим без справочника):

```
subject_code = "СУБЪЕКТ-" + sha256(str(lead.id)).hexdigest()[:16]
# str(lead.id) — канонический UUID в нижнем регистре с дефисами (как отдаёт asyncpg/psql).
```

- **16 hex = 64 бита** (правка spec-review: 10 hex = 40 бит давали ~0.1% коллизий на row_cap=50k → неверная
  деобфускация). На масштабе тенанта коллизий практически нет; `map.csv` дополнительно ASSERT'ит уникальность
  `subject_code` в пределах выгрузки (fail-closed при коллизии).
- **Без соли** — детерминизм важнее, файл и так необратим без `map`. (HMAC с per-tenant солью — возможная
  будущая мера, не нужна для №140.) Один лид = один `subject_code` во всех выгрузках, стабильно между запусками.

## Отношение к существующим экспортам (правка spec-review)
Уже есть: `/export.csv` (`export_masked` → `db.stream_export_masked`) — ⚠️ отдаёт ФИО (`name`) ОТКРЫТЫМ
текстом, маскирует лишь 2 цифры телефона → это операционный экспорт тенанта (законный доступ оператора к ФИО
СВОИХ лидов), **НЕ обезличивание**; и `/export-full.csv` (`export_full`, gated: `confirm=yes` +
`_has_narrowing_filter`) — полные телефоны.
- Новый `anon.csv` — это **НЕ «masked»**: `name` и все прямые id заменены на `subject_code`. Отдельный стрим
  оправдан (другой набор колонок + псевдоним + `has_notes`), но WHERE/limit строим переиспользуя паттерн
  (`pool.acquire`+`c.transaction`+`c.cursor`, row_cap как `stream_consent_events`); фильтров нет — вся база
  тенанта (RLS-скоуп).
- В UI раздел «Защита данных» отделён от операционного «Экспорта», чтобы тенант не путал, какой файл —
  обезличенный для регулятора. Поведение существующего `/export.csv` НЕ меняем.

## Компоненты (всё в `admin-panel`, БЕЗ DDL)
Новый раздел кабинета **«Защита данных (152-ФЗ)»** (`/data-protection`), скоуп — активный тенант (RLS,
видит только свою базу). Сюда же позже ляжет «Справка о ПД» (#3). Nav-пункт виден тенанту (вне `is_platform`).

**UI/Nav:**
- `nav_item('data_protection', '/data-protection', 'Защита данных', active)` — разместить ВНЕ блока
  `{% if session.is_platform %}` (рядом с `lead_magnet`/`nurture` в `base.html`).
- Добавить ветку `data_protection` в макрос `nav_icon` (lucide `shield`/`lock`) — иначе пункт без иконки
  (макрос без `else`-fallback). Добавить метку в словарь меток `base.html`.
- Новый шаблон `admin-panel/templates/data_protection.html`: краткое пояснение (что такое обезличивание/№140) +
  две POST-формы с CSRF (как кнопки экспорта). GET-страница — отдельный роут `/data-protection`.

**Действие 1 — «Скачать обезличенную базу»** (основное, self-serve):
- `POST /data-protection/anon.csv` → `StreamingResponse` CSV (UTF-8 + BOM, `_csv_line(..., bom=True)`).
- `db.stream_leads_anon(*, row_cap)` — курсорный стрим под RLS; **explicit whitelist SELECT** колонок «остаются»
  + `id` (для псевдонима). Псевдонимизация (`subject_code`, `has_notes`, валидация `ai_persona`) — в Python.
- **RLS defence-in-depth:** в стриме явный `set_config('app.tenant_id', $tid, true)` внутри транзакции
  (не полагаться только на pool-setup-хук) + guard: если активный тенант пуст → `raise` (не отдавать
  неопределённый/пустой набор молча). Инвариант — комментарием у функции.
- **Аудит ДО стрима** (fail-closed): `db.audit(action='pii_export_anon', detail={matched, row_cap})` —
  БЕЗ ПДн. `matched` = `count_leads` по тенанту (как делают существующие экспорты).
- **row_cap = `config.EXPORT_ROW_CAP` (50k).** Если `matched > row_cap` — отдать явный маркер неполноты
  (строка-маркер в хвосте CSV и/или предупреждение в UI), чтобы регулятору не ушёл молча усечённый набор.

**Действие 2 — «Скачать справочник соответствия»** (ОТДЕЛЬНОЕ, усиленный гейт):
- `POST /data-protection/map.csv` — CSV `subject_code → name, phone, tg_user_id, vk_user_id, max_user_id,
  max_chat_id, web_session_id, notes, erase_status` (raw — это защищённый справочник, реверс псевдонима;
  `erase_status` — последняя колонка, флаг отзыва согласия).
- **Гейтинг не слабее `export_full`** (правка spec-review — справочник чувствительнее): обязательный
  `confirm == "yes"` (отдельный чекбокс «понимаю: это реверс псевдонима, храню отдельно») + поле формы
  **«повод/основание»** (непустое) + `_enforce_csrf`. Ограничение по роли (только владелец, не любой оператор) —
  пометить TODO (текущая модель ролей ограничена).
- **Окно отзыва согласия** (правка spec-review): для лидов с непустым `erase_requested_at` в `map.csv` —
  `name/phone/notes` обнуляем + флаг `erase_status='отзыв — обезличивание в процессе'` (не выдавать реверс ПДн
  субъекта, потребовавшего прекращения; в `anon.csv` он идёт псевдонимизированным — это допустимо).
- `db.stream_leads_map(*, row_cap)` — тот же курсорный/RLS-паттерн + те же defence-in-depth-гарды; ASSERT
  уникальности `subject_code` (коллизия → fail-closed).
- **Аудит ДО стрима:** `db.audit(action='pii_export_map', detail={matched, row_cap, reason})` (повод пишем,
  ПДн — НЕТ). Отдельная запись «передача справочника».
- В UI: предупреждение — хранить отдельно от обезличенного набора, не объединять, отдавать только по отдельному
  законному запросу.

**Формат/безопасность CSV:**
- CSV per действие; CSRF на POST; стрим курсором (как `stream_consent_events`).
- **Formula-guard (правка spec-review, CSV-инъекция):** `_csv_line` (или формирующий слой) нейтрализует ведущие
  `= + - @ TAB CR` в текстовых полях из пользовательского ввода (`name`/`notes`/`phone`) — префикс апострофа.
  Заодно закрывает существующие `export_masked`/`export_full`. Покрыть тестом.

## Тесты
- **Чистый смоук** `scripts/pii_anon_smoke.py` (без БД): детерминированность и длина `subject_code`
  (16 hex, стабильность между вызовами), уникальность на наборе id; `has_notes`-логика; валидация `ai_persona`
  против `PERSONA_PRESETS`; **formula-guard** (`=cmd`/`+`/`-`/`@`/таб/CR → нейтрализуются); инвариант
  «прямые идентификаторы НЕ присутствуют в наборе колонок anon».
- **БД-смоук** `scripts/pii_anon_db_smoke.py` на `risuy_dev` (owner-DSN inline из Timeweb API, VPN выкл):
  throwaway-тенант + лиды (вкл. лид с `erase_requested_at` и лид с ФИО/адресом в `notes`) →
  `stream_leads_anon` не отдаёт ни одного прямого идентификатора и raw-`notes`; `stream_leads_map` отдаёт
  соответствие; для отозвавшего лида `map` обнуляет `name/phone/notes` + флаг; round-trip `subject_code`
  стабилен и совпадает между anon/map; RLS режет чужой тенант (0 строк). ⚠️ Чистка throwaway: FK без cascade —
  удалять `leads` перед тенантом (как в прошлых смоуках).

## Решённые в spec-review микровопросы (дефолты применены)
- Код субъекта — детерминированный `sha256(str(lead.id))[:16]`, без соли. (Альтернатива — сквозная нумерация
  отвергнута: нестабильна между выгрузками.)
- `notes` — исключён из обезличенного файла (только `has_notes`), сырой — лишь в gated `map.csv`.
- Лиды в окне отзыва (`erase_requested_at`) — в `map.csv` ПДн обнулены, в `anon.csv` псевдонимизированы.
- Псевдоним 16 hex (коллизии); formula-guard CSV; усиленный гейт `map.csv`; RLS defence-in-depth в стримах.
