# Дизайн #1 — Обезличенная выгрузка базы тенанта для РКН/НСУД (Приказ №140)

**Дата:** 2026-06-28 · **Статус:** дизайн согласован, ждёт реализации (новый чат: spec-review → writing-plans → impl).
**Часть плана 152-ФЗ из 3 кусков** (делаем по порядку): **#1 эта выгрузка** → #2 YandexGPT-бэкенд + тумблер →
#3 «Справка о ПД» в кабинете + публичная страница.

## Зачем
По Приказу РКН №140 (с 01.09.2025) + ЕИП НСУД: Минцифры может потребовать **обезличить** базу ПДн и передать
в систему. Если у тенанта нет пайплайна обезличивания — требование не выполнить → штраф (ось 1 «способность
обезличить по требованию», не зависит от места инференса). Цель: тенант **одной кнопкой** отдаёт обезличенный
набор своей базы клиентов. Это и есть «спать спокойно» по статье (habr selectel/943422).

## Метод (Приказ №140, Приложение №2 — «введение идентификаторов» / псевдонимизация)
Прямые идентификаторы заменяются на стабильный код субъекта; отдельно — защищённый справочник соответствия.

## Объём v1
**База ЛИДОВ** (`leads`) активного тенанта. Диалоги (`messages`, свободный текст) — ОТДЕЛЬНЫЙ инкремент
(нужен NER). YAGNI-граница ради скорости и точности #1.

**Поля `leads`** (по схеме):
- Прямые идентификаторы → заменить/убрать: `name`, `phone`, `phone_hash`, `tg_user_id`, `vk_user_id`,
  `max_user_id`, `max_chat_id`, `web_session_id`. Свободный `notes` → прогнать через `shared/pii` (телефон/
  email/ИНН), пометить «обезличено частично» (свободный текст — остаточный риск, как в `152fz-legal-review-checklist.md`).
- Остаются (не идентифицируют, полезны регулятору): `subject_id`, `messenger`, `source`, `consent`,
  `subscribed`, `status`, `created_at`, `updated_at`, `guide_sent_at`, `follow_up_1..3_at`, `unsubscribed_at`,
  `erase_requested_at`, `ai_persona`. (`survey` — пусто, не включаем.)

## Код субъекта
Детерминированный из `lead.id` (uuid): один лид = один `СУБЪЕКТ-<код>` во всех выгрузках; односторонний
(обезличенный файл сам по себе необратим без справочника). Алгоритм оператора — допустимо по №140.
Решение: `СУБЪЕКТ-` + первые ~10 hex sha256(lead.id) (стабильно, без коллизий на масштабе тенанта).

## Компоненты (всё в `admin-panel`, БЕЗ DDL)
Новый раздел кабинета **«Защита данных (152-ФЗ)»** (`/data-protection`), скоуп — активный тенант (RLS,
видит только свою базу). Сюда же позже ляжет «Справка о ПД» (#3). Nav-пункт виден тенанту (вне is_platform).

**Действие 1 — «Скачать обезличенную базу»** (основное, self-serve):
- `POST /data-protection/anon.csv` → стрим CSV (UTF-8 + BOM, как `consents.csv`).
- db: `stream_leads_anon(row_cap)` под RLS → строки лидов; псевдонимизация в Python (subject_id + маскер для notes).
- Аудит: `admin_audit action='pii_export_anon'` ДО стрима (факт выгрузки).

**Действие 2 — «Скачать справочник соответствия»** (ОТДЕЛЬНОЕ, gated):
- `POST /data-protection/map.csv` — CSV `subject_id → name/phone/channel-ids`.
- В UI: отдельная кнопка с предупреждением (хранить отдельно, отдавать только по отдельному законному запросу).
- Аудит: `admin_audit action='pii_export_map'` (отдельная запись, кто/когда). Можно потребовать доп. подтверждение (чекбокс/повторный CSRF).

**Формат:** CSV per действие; CSRF на POST; стрим курсором (как `stream_consent_events`).

## Тесты
- Чистый смоук: псевдонимизатор (детерминированность subject_id; маскер notes; идентификаторы не утекают в обезличенный набор).
- БД-смоук на risuy_dev: throwaway-тенант+лиды → `stream_leads_anon` не отдаёт прямые идентификаторы; `stream_leads_map` отдаёт соответствие; round-trip subject_id стабилен.

## Вне #1
YandexGPT-бэкенд (#2); «Справка»/публичная страница (#3); экспорт диалогов `messages` (нужен NER); НСУД-формат API (v1 — структурный CSV).

## Открытый микровопрос (дефолт выбран)
Код субъекта — детерминированный из `lead.id` (рекомендовано). Альтернатива — сквозная нумерация «Субъект ID-1,2,3…»; детерминированный проще/стабильнее, берём его, если владелец не возразит.
