## 🆕 «Маркетинг-штаб» → ЯДРО: SEO-Контент-движок (лидоген INBOUND)
**Решение владельца:** это ядро модуля «Маркетинг-штаб» (не отдельная сущность); маркетинг-штаб = зонтик (SEO-движок + генерация постов/кампаний).
**Источник:** рил iliakud (17.07.2026). Суть: контент на реальных поисковых запросах → входящий поток заявок.

### Что делает
1. **Анализ ниши через Яндекс Вордстат** — тянет НЧ + ВЧ запросы по нише тенанта.
2. **Контент-план** — раскладывает запросы в график выхода статей (приоритет по частотности/конкуренции).
3. **ИИ-генерация SEO-статей** — по каждому запросу статья с вшитым SEO + ссылками на продукты/услуги тенанта.
4. **Автопостинг** — блог + каналы (TG/VK; Instagram — под вопросом из-за геоблока РФ) по расписанию.
5. **ИИ-ответы под запросами** — отвечать на живые вопросы/комментарии под контентом.

### Переиспользуем существующее (снижает объём)
- База знаний + продукты тенанта → источник фактов и ссылок в статьях.
- Автопостинг в TG/VK-каналы (broadcast, VK posting) — уже есть.
- Расписание (user_preferences, миграция 0011) — уже есть.
- LLM-шлюз (генерация) — тот же.

### Новое (построить)
- Интеграция **Яндекс Вордстат** (официальный API vs парсинг — решить; лимиты/цена).
- **Блог-движок** (страницы статей на сайте тенанта / поддомене) под SEO.
- Планировщик контент-плана из запросов.

### Биллинг (токен-пул) — черновые строки прайса

## 🧭 Методология лид-движка (OUTBOUND) — по first-customer-finder
**Источник:** github.com/Kappaemme-git/codex-first-customer-finder-skill (изучено 20.07.2026). Берём **методологию**, не файл-скилл (он под Codex). Ложится на движок лидов (коллекторы TG/VK → S4-классификатор → matching → лиды) и валидирует уже построенные SL-провенанс + HumanGate.

### Что заимствуем
1. **ICP сначала** (из брифа тенанта): основной + смежный профиль, триггеры боли, дисквалификаторы. Единый вход и для inbound (Вордстат), и для outbound (сигналы).
2. **5 корзин сигналов → таксономия S4-классификатора** (regex/словарь + vLLM): явный спрос · боль · воркэраунд · переключение · тайминг.
3. **Скоринг 5 осей (0–100) → рубрика `engine.matching`:** боль 25% · соответствие 25% · тайминг 20% · достижимость 15% · качество доказательства 15%. Пороги: 80+ сильный / 65–79 проверить / 50–64 слабый / <50 отсекать.
4. **Стадии лида:** high-intent · problem-aware · trigger-present · potential-fit.
5. **Реестр доказательств + ссылка на источник** = наш **SL-провенанс** (`outbound_signal`, ссылка на сообщение, `consent=false`).
6. **Черновик захода, ручная отправка** (4 части: контекст→боль→продукт в 1 фразу→1 вопрос, <90 слов) — совпадает с **HumanGate + fail-closed**; НЕ автоотправка.
7. **Формат лид-отчёта тенанту** (вердикт + шорт-лист + повторяющиеся паттерны + 7-дневный ручной план) — UX-выдача движка в кабинете.
8. **Приватность** — у нас строже: 152-ФЗ + РКН-анонимизация.

### Адаптация под нас
- **Слой источников — НАШ:** TG/VK-коллекторы + биржи/тендеры (не общий веб/LinkedIn их скилла). Таксономия сигналов и скоринг переносятся напрямую.
- Портируем в **промпты движка** (классификатор/скорер) ИЛИ в собственный Claude Code-скилл — не как Codex-файл.

### Куда вписать
→ в спеку **S4-классификатора** (таксономия + скоринг) и **matching**; → в **лид-отчёт кабинета**. Всё — отдельная спека движка лидов, **вне** токен-биллинг-спеки.
