# Дизайн-спек: per-tenant чек 54-ФЗ (ДИЗАЙН, кода нет)

Статус: **СПЕК. Кода на денежном пути НЕ написано.** Реализация — отдельной волной с тем же ригором,
что W2 (схема-first → код → смоук → адверсариальное ревью → тест на ЮKassa-тест-магазине → dev→прод).
Playbook (Launch): compliance — «required remediation, не suggestion», и continuous workstream.

## 1. Проблема (как сейчас в коде)

`bot-telegram/yookassa.py::create_payment` прикладывает чек **только для магазина Школы**
(env `SHOP_RECEIPT_ENABLED`/`SHOP_VAT_CODE`), и ЯВНО НЕ прикладывает для кассы тенанта:

```py
# create_payment(...): строки 88-95
if creds is None:                       # ← только магазин Школы (env)
    receipt = _receipt(lead_phone, ...) # один subject 'service', vat_code из env
    if receipt is not None: body["receipt"] = receipt
# creds задан (магазин ТЕНАНТА) → чек НЕ прикладывается вовсе
```

Следствие: когда лид платит на магазин **тенанта** с включённой фискализацией (ОФД), ЮKassa может
**отвергнуть платёж без чека** → оплата зависает; либо чек не выбивается (нарушение 54-ФЗ у тенанта).
Кросс-тенант фиск-утечку (чужой ОФД/НДС) код уже не допускает — школьный чек к тенанту НЕ цепляется.

Структура чека ЗЕРКАЛИТСЯ в 4 билдерах (`bot/yookassa._receipt`, `panel/yookassa._shop_receipt`,
`app._service_receipt`, `_recurrent_receipt`) — дрейф полей ломает фискальный магазин.

## 2. Цель / НЕ-цель

- **Цель:** при оплате на магазин ТЕНАНТА прикладывать чек 54-ФЗ, построенный из **фискального
  конфига ТЕНАНТА** (его НДС/налоговый режим/контакт покупателя), а не из env Школы.
- **НЕ-цель (v1):** возвратные чеки (refund), маркировка товаров, агентские/комиссионные чеки сложных
  схем — отдельно (см. §7). Голос/нефинансовые потоки — не трогаем.

## 3. Требования

1. **Per-tenant фискальный конфиг:** `enabled`, `vat_code` (ставка НДС), `tax_system_code` (СНО —
   обязателен, если у магазина несколько систем налогообложения), `payment_subject` (предмет расчёта:
   service/commodity), `contact_policy` (чек на телефон лида / email / запросить).
2. **Tenant-aware билдер чека:** ЕДИНЫЙ хелпер `build_receipt(fiscal_cfg, contact, description, amount)`
   — НЕ дублировать пятый билдер; по возможности свести существующие к одному (анти-дрейф).
3. **Контакт покупателя:** 54-ФЗ требует телефон ИЛИ email. Источник: `leads.phone` (есть у части),
   email — собрать шагом перед оплатой ИЛИ из профиля. Нет контакта + fiscal включён → §6.
4. **Идемпотентность/безопасность:** не менять существующий путь Школы (env) и by_payment-матч; чек —
   аддитивно к телу платежа; Idempotence-Key=order.id сохраняется.

## 4. Данные (вероятно БЕЗ DDL)

Хранить в `tenant_settings` (key-value, как `online_payments_enabled`/AI-ключи) — **новых колонок не
нужно, DDL не требуется**:

| ключ | значение | смысл |
|---|---|---|
| `fiscal_enabled` | `""`/`"1"` | прикладывать ли чек к платежам тенанта |
| `fiscal_vat_code` | `1`..`6` | ставка НДС (1 = без НДС, типично УСН/самозанятый) |
| `fiscal_tax_system_code` | `1`..`6` или `""` | СНО (обязателен при нескольких на магазине) |
| `fiscal_payment_subject` | `service`/`commodity` | предмет расчёта |
| `fiscal_contact` | `phone`/`email` | откуда брать контакт покупателя |

Панель: блок «Фискализация» в «Продукты»→«Приём оплаты» (рядом с shop_id/ключом). Грант panel_rw —
table-level на tenant_settings уже есть (граблей column-INSERT нет).

## 5. Точка внедрения (минимальный диф)

- `create_payment(...)`: добавить параметр `tenant_fiscal: dict | None`. Логика:
  `if creds is None: <как сейчас, Школа env>` ; `elif tenant_fiscal and tenant_fiscal["enabled"]:
  body["receipt"] = build_receipt(tenant_fiscal, contact, description, amount)`.
- Вызовы создания заказ-платежа (3 точки: `handlers.py:240`, `multiplex.py:201`, `app.py:1430`):
  подгрузить фискальный конфиг тенанта (из tenant_settings, как get_tenant_ai_overrides) + контакт лида,
  передать в create_payment. Школьный путь (creds=None) НЕ трогаем — байт-в-байт.

## 6. Edge-кейсы (требуют решения)

- **Нет контакта покупателя + fiscal_enabled:** (а) не прикладывать чек + лог + мягкий фолбэк лиду
  (текущее поведение «без чека»); ИЛИ (б) шаг сбора email перед оплатой. → решение владельца (а проще/безопаснее).
- **Магазин тенанта без фискализации:** `fiscal_enabled=""` → чек не прикладываем (как сейчас) — ОК.
- **СНО не задан, а у магазина несколько:** ЮKassa отвергнет → валидация в панели (требовать tax_system).
- **Дрейф структуры:** свести к одному билдеру + смоук, проверяющий идентичность полей всех путей.
- **Возврат (refund):** 54-ФЗ требует возвратный чек — В v1 НЕ входит, явно отметить как долг.

## 7. Открытые вопросы для владельца (бизнес/юр — нужны ДО реализации)

1. **Кто фискальный субъект** — тенант (его касса/ОФД, его чек) или платформа (агентская схема)?
   От этого зависит, агентский ли это чек 54-ФЗ или прямой чек тенанта.
2. **У каждого ли тенанта свой ОФД/касса с фискализацией** в ЮKassa, или большинство без неё?
3. **Система налогообложения тенантов** (УСН/ОСН/самозанятый/патент) → дефолты vat_code/tax_system_code.
4. **Контакт покупателя:** опираемся на `leads.phone` (есть не у всех) или собираем email?
5. **Возвраты** — нужны ли возвратные чеки в этой волне?

## 8. План реализации (когда ответишь на §7)

1. Схема: новые ключи tenant_settings — DDL НЕ нужен (key-value). Панель: блок «Фискализация».
2. Код: единый `build_receipt` + `tenant_fiscal` в create_payment + подгрузка в 3 точках. Школьный путь не трогаем.
3. Смоук: структура чека per-tenant (vat/tax/subject/контакт); негатив (нет контакта → не падаем).
4. Адверсариальное ревью (денежный путь!) + тест на ЮKassa **тест-магазине с фискализацией**.
5. Раскатка dev→прод; ⚠️ неверный чек → магазин отвергает платёж → оплаты зависают (как с RLS — порядок и тест важны).
