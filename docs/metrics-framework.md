# Measurement-framework risuy (Launch-стадия)

Источник идеи: Anthropic «The Founder's Playbook», глава Launch — *«establish your
measurement framework before the first user shows up»* и *«make the adversarial case against
your own traction»*. Цель — знать и защищать метрики до масштабирования, а не подбирать
их задним числом под удобную картинку.

**Чем считаем:** `scripts/metrics_report.py` — read-only снимок (только `SELECT` + пояс
`default_transaction_read_only=on`; объекты в БД НЕ создаёт). Запуск:

```bash
METRICS_DSN="postgresql://gen_user:<pw>@81.31.246.136:5432/risuy?sslmode=require" \
    ./.venv-smoke/bin/python scripts/metrics_report.py     # ПРОД (реальные данные), read-only
```
Пароль owner-DSN получать inline из Timeweb API (как в смоуках), в файл не писать.

## Два уровня метрик

risuy — **B2B-реселлинг**: тенант = платящий клиент, который запускает свою Лию на своих лидах.
Поэтому два контура:

1. **ПЛАТФОРМА** — бизнес самого risuy: привлечение / активация / удержание / выручка тенантов.
2. **ПРОДУКТ** — ценность, которую Лия приносит тенанту (воронка лидов → оплата). Это и прокси
   «липкости»: тенант, у которого Лия конвертит лидов, не уходит.

## Определения метрик

| Метрика | Как считается (таблица/поле) | Почему (playbook) |
|---|---|---|
| **Тенанты всего / active** | `tenants` (status) | база привлечения |
| **Новых за 7/30д** | `tenants.created_at` | темп привлечения по когортам |
| **Активирован тенант** | есть лид с ≥1 **входящим** сообщением (`messages.direction='in'`) | activation, канал-агностично, по реальному использованию (не по «настроены секреты») |
| **Удержание (30д)** | тенант с ≥1 `messages` за 30д | retention — «продукт тянет, а не толкаешь» |
| **Платформенная выручка** | `service_invoices(status=paid).amount`, `subscriptions(status=active)`, `usage_ledger.charged_microrub` | MRR/потребление — основа CAC/LTV/payback |
| **Воронка лидов** | `leads.status`: new→guide_sent→nurturing→converted; флаги `escalated_at`, `unsubscribed_at` | где лиды отваливаются |
| **Конверсия лид→оплата** | `orders(status=paid)` / `leads` | главный продуктовый исход |
| **Средний чек** | `avg(orders.amount) where paid` | вход в LTV |
| **Активные лиды (7д)** | `messages(direction=in, 7д)` distinct lead | вовлечённость |

### Целевые бенчмарки (ГИПОТЕЗЫ — калибровать по первым данным)
Playbook требует задать цели *до* прихода данных. Сейчас данных нет → это стартовые гипотезы H,
а не факт; пересмотреть после первых 10–20 реальных тенантов.

- **H1 — активация тенанта за 7 дней ≥ 60%** (подписался → получил первого вовлечённого лида).
- **H2 — удержание тенантов 30д ≥ 70%** (активность за последние 30д).
- **H3 — конверсия лид→оплата: калибровать** (нет данных; ориентир для тёплой воронки 3–8%).
- **H4 — доля лидов без вовлечения < 30%**.

## 🚩 Tripwires (сигналы ложного PMF)

Playbook: заранее определить, как выглядит false-positive. Если триггер сработал — это сигнал
*не строить дальше, а разобраться*.

| Tripwire | Порог тревоги | Что значит |
|---|---|---|
| Тенанты подписались-но-НЕ-активировались | > 40% | проблема онбординга/активации, а не привлечения |
| Тенанты активны-но-уснули (нет msg 30д) | > 30% | проблема удержания — «толкаешь, а не тянет» |
| Лиды без вовлечения (нет входящих) | > 30% | привлечение трафика ≠ интерес; проблема оффера/канала |
| Выручка без активности 30д | > 0 тенантов | «revenue without retention» — самый коварный false-PMF |

## Как пользоваться

- Запускать **еженедельно** (и перед/после любого изменения воронки), фиксировать тренд.
- Когда появятся реальные тенанты — пересмотреть гипотезы H1–H4 на фактические бенчмарки.
- При срабатывании tripwire — диагностика (где именно отваливается), а не новые фичи.
- Дальше (по желанию, отдельная безопасная волна): промоутнуть метрики в админ-дашборд панели.
  ⚠️ Если делать VIEW над RLS-таблицами — НЕ давать `panel_rw` доступ к owner-овым вью (иначе
  кросс-тенант-утечка в обход RLS); аналитика — строго owner/платформа-сайд.
