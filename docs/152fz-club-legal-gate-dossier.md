## 0. TL;DR — гейт в двух слоях

«Клуб» — opt-in-разворот вместо холодного парсинга; легальный базис = согласие; красная линия = ни одного холодного контакта. **Красная линия по коду ДЕРЖИТСЯ** (см. §5). Но гейт закрывают ДВА слоя дефектов:

- **СЛОЙ 1 — тексты согласий** (состав ст.9 152-ФЗ + ФЗ-38 + intro-раскрытие): клубные согласия тоньше эталона воронки; intro фиксирует хэш текста, который субъект не видел. Правки — код+тексты, **без DDL**. Почти всё схлопывается в «поднять `build_club_consent_text` до эталона `build_consent_text`».
- **СЛОЙ 2 — функциональные 152-ФЗ дыры** (НЕ про тексты): задеплоенный клуб не может исполнить **отзыв согласия** (ст.9 ч.2) и **срок хранения** (Политика §6.4) для клуб-членов. Живая экспозиция с первого реального члена. Требует малого DDL + функций.

## 3. ТРЕК B — двусторонний intro (знакомство)  `shared/club.py:31-33` + `handlers.py:215-589`

Двусторонняя механика верифицирована и **корректна** (§5). Дефекты — в тексте согласия:
- 🔴 **intro-scope / L3-underdescribes (critical):** согласие обещает абстрактный «обмен контактами», а карточка `club_intro_contact_card` реально отдаёт второй стороне: название бизнеса, город, ОКВЭД, **имя контактного лица (физлицо), телефон**. Субъект соглашается, не зная состав раскрытия. (Уточнение верификатора: ИНН в карточку не рендерится — критичны имя+телефон.)
- 🔴→important **intro-hash-mismatch / shown≠hashed:** субъекту показан `CLUB_INTRO_OFFER`, а `text_hash` считается над `build_club_consent_text('intro')` — строкой, которую он не видел. Доказательная база в consent_events некорректна. (club_join как контрпример делает правильно.)
- important **intro-operator:** intro-согласие не называет оператора и ИНН.
- important **L3-intro-policy-link-absent:** ни в тексте, ни в UI intro нет ссылки на Политику.

**Предлагаемый текст — см. `152fz-club-consent-proposed-texts.md §2`** (показывается = хэшируется; перечисляет состав раскрытия; называет оператора; порядок отзыва).

## 6. 🔴 ФУНКЦИОНАЛЬНЫЕ ДЕФЕКТЫ 152-ФЗ (СЛОЙ 2 — НЕ про тексты; до-проверены вручную)

- 🔴 **L4-revoke-noop-club (critical):** `cmd_revoke → request_erase` (db.py:141-162) трогает ТОЛЬКО `leads`; `consent_events('revoked')` пишется лишь при найденной строке leads. Чистый клуб-член (вход `?start=club`, `lead_id=None`, без `upsert_start`) → строки leads нет → `/revoke` = тихий no-op: ни обезличивания, ни события отзыва. **Обещание club_join-текста «отозвать можно в любой момент» в коде невыполнимо.** У члена с `lead_id=None` вообще нет обратной связи `tg_user_id → член`.
- 🔴 **L4-retention-club-tables (critical):** retention-cron (`due_for_erase`/`erase_lead`, db.py:2037-2105) обезличивает только leads/messages/agent_memory/link_clicks/… — **никогда** `club_members`(display_name/city/okved/inn)/`club_profiles`(offering/seeking/description). Клубные бизнес-ПДн живут бессрочно даже после отзыва → нарушение срока хранения (Политика §6.4 «≤30 дней»).
- important **L4-policy-global-not-tenant:** клубная кнопка Политики ведёт на глобальный `config.PRIVACY_URL` (единый bot-env для ВСЕХ тенантов), не на пер-тенантную `/legal/{slug}/privacy` — воронка уже умеет пер-тенантную. Мульти-тенант: показанная Политика может быть чужого оператора.
- important **L4-policy-missing-club-categories:** `build_privacy_policy §3.1` перечисляет только «имя»/«имя и телефон». Клубные категории (название бизнеса, город, ОКВЭД, цепочка, offering/seeking, контакты при intro) НЕ покрыты — нужен режим «клуб» в Политике.
- important **L4-registry-no-club-revoked:** нет записи события `revoked` для клуба (только `granted`) → нет append-only доказуемости отзыва.

**Правки СЛОЯ 2 (требуют малого DDL + функций):** канал идентичности в `club_members` (tg/vk/max_user_id) + `erase_requested_at`/`offers_opt_in`; `cmd_revoke` ищет членство и ставит пометку + пишет `revoked`; новый `club_due_for_erase`/`club_erase_member` (обезличивает in-place, НЕ CASCADE); расширить `erase_lead` для промотированных членов; режим «клуб» в `build_privacy_policy`.
